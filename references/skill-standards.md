## 检查清单

### 1. 前言区完整性

```yaml
---
name: {kebab-case, ≤50字符}
version: v{语义化版本}
author: {作者名}
description: {100-150字符，一句话描述}
tags: [{5-15个标签}]
dependency:
  parent: {父技能名 或 none}
  children: [{子技能列表}]  # 可选
---
```

| 检查项 | 通过标准 |
|--------|---------|
| name | kebab-case，≤50 字符 |
| version | vX.Y.Z 格式 |
| description | **100-150 字符**（硬性约束） |
| tags | 5-15 个 |
| parent | 指向父技能或 `none` |

#### description 编写规则（CSO 优化）

> **核心原则**: description 只写触发条件，不写工作流总结。防止 Agent 走捷径跳过正文。

| # | 规则 | 说明 | 好例子 | 坏例子 |
|---|------|------|--------|--------|
| 1 | **以 "Use when..." 开头** | 聚焦触发条件 | "Use when creating new skills..." | "技能创建指南和设计模式库" |
| 2 | **只写触发条件** | 不总结工作流或步骤 | "Use when tests have race conditions" | "用子代理执行任务并审查代码" |
| 3 | **具体症状** | 用户实际遇到的问题 | "技能不如预期、Agent 绕过规则时" | "需要技能创建时使用" |
| 4 | **关键词覆盖** | Agent 可能搜索的词 | "skill / 技能 / SKILL.md / agent / TDD" | 单个术语 |
| 5 | **第三人称** | 注入系统提示 | "Use when encountering any bug..." | "I can help you debug..." |
| 6 | **长度控制** | <500 字符（最好 <200） | 简洁的触发条件列表 | 详细的功能描述段落 |
| 7 | **写用户会问什么** | 描述什么请求会触发技能 | "用户请求代码审查时触发" | "本技能处理代码审查工作" |
| 8 | **避免歧义** | 两个技能 description 相似会导致随机激活 | 每个技能描述有辨识度 | 通用模糊描述 |

> 💡 **CSO 核心理念**：description 不是摘要，是**触发信号**。它在 Agent 启动时决定哪些技能被激活。
>
> 📖 **详细规则**: [./writing-rules.md](./writing-rules.md) — R9: CSO Description 编写规则

### 2. 必备章节

| 章节 | 说明 |
|------|------|
| **任务目标** | 一句话说清技能做什么 |
| **触发条件** | 什么情况下使用此技能 |
| **操作步骤** | 清晰的操作流程 |
| **示例/场景** | 至少 1 个使用示例 |
| **注意事项** | 边界条件、前置依赖 |

### 3. 命名规范

**轻量工坊模式**（推荐）：

| 对象 | 规范 | 示例 |
|------|------|------|
| 目录名 | kebab-case | `skill-factory-creator` |
| 技能文件 | `SKILL.md`（大写） | `SKILL.md` |
| 阶段指南 | `{skill}-{阶段名}` | `skill-factory-creator` |
| 标签 | kebab-case 小写 | `three-layer-architecture` |

**完整工厂模式**（大项目）：

| 对象 | 规范 | 示例 |
|------|------|------|
| 协调器 | `{skill}-{domain}` | `skill-factory-production` |
| 执行者 | `{skill}-{worker}` | `skill-factory-code-checker` |

### 4. 链接有效性

- ✅ 所有内部链接指向存在的文件
- ✅ 使用相对路径（`../`、`./`）
- ✅ 锚点链接与实际标题匹配
- ✅ 不引用 `docs/` 目录（已废弃）

### 5. 层级合规

```
计算规则：
- 从 SKILL.md 所在目录开始计数
- 每深入一级子目录 +1
- references/ scripts/ 不算层级
- 最大深度: 3 层

✅ 轻量: skill/SKILL.md → skills/creator/SKILL.md (2层)
✅ 完整: skill/SKILL.md → skills/production/SKILL.md → skills/production/code-checker/SKILL.md (3层)
❌ skills/production/code-checker/sub/SKILL.md (第4层！)
```

### 6. 内容质量

| 维度 | 要求 |
|------|------|
| **可读性** | 使用表格、Mermaid 图表辅助 |
| **可操作性** | 每个步骤有明确的输入输出 |
| **完整性** | 覆盖主要的成功和失败路径 |
| **简洁性** | 核心流程 <300 行，详细信息放 references/ |

## 评分体系

| 维度 | 权重 | 检查点 |
|------|------|--------|
| 前言区 | 25% | name/version/description/tags/parent |

## 写作质量检查

> 高质量的 SKILL.md 不仅通过格式检查，还要通过写作质量关。以下是进阶检查项。

### 注意事项升级要求

每个技能的"注意事项"章节不应只是泛泛提醒，应至少包含以下 **三类内容之一**：

| 类型 | 要求 | 示例 |
|------|------|------|
| **Gotchas 坑点** | ≥3 个具体环境陷阱 | "查询用户表必须加 WHERE deleted_at IS NULL" |
| **错误处理矩阵** | 按异常类型分类的处理表 | 输入错误/工具失败/数据异常 各有对应操作 |
| **反模式列表** | 每个"不要"配"改为" | ❌ 泛泛指令 → ✅ 具体异常分类+处理方式 |

### 反模式速查

| 反模式 | 为什么失败 | 正确做法 |
|--------|-----------|---------|
| "适当处理错误" | Agent 不知道什么叫"适当" | 按 5 类异常分别定义处理方式 |
| "注意格式" | 太模糊，每次理解不同 | 给出具体格式模板或示例 |
| "根据情况决定" | Agent 的判断与你不一致 | 给出决策树或判定条件表 |
| 只写"不要 X" | Agent 无所适从，不知道该做什么 | "不要 X，因为 Y；改为 Z" |

> 📖 完整写作规则: [./writing-rules.md](./writing-rules.md)

## 常见问题

| 问题 | 解决方案 |
|------|---------|
| description <100 字符 | 补充技能的核心能力和适用场景 |
| description >150 字符 | 删除冗余修饰词，聚焦核心价值 |
| 目录名含大写 | 改为全小写+连字符 |
| 子技能 >3 层 | 拆分或使用 references/ |
| 缺少必备章节 | 补充操作步骤和示例 |
| description 歧义模糊 | 加入触发关键词和适用场景，确保与同类技能可区分 |
| SKILL.md >500 行 | 将详细参考信息移至 references/，主体保留核心流程 |
| 子技能间引用死链 | 检查所有相对路径是否指向实际存在的文件 |
| 注意事项太泛泛 | 升级为 Gotchas/错误处理/反模式三类具体内容，参考 writing-rules.md |
| 缺少错误处理章节 | 按输入/工具/数据/权限/超时 5 类分别定义处理方式和反馈信息 |
| 操作步骤缺少验证 | 每个关键步骤后追加二进制验证项（可客观判断通过/不通过） || 给了选项而非默认值 | Agent 不擅长选择，提供推荐默认值，仅在必要时列替代方案 |

## TDD 验证检查（v6.0 新增）

> **铁律**: `NO SKILL WITHOUT A FAILING TEST FIRST`
>
> 每个技能发布前必须经过 TDD 验证，或明确说明豁免原因。

### 检查清单

| # | 检查项 | 通过标准 | 豁免条件 |
|---|--------|---------|---------|
| 1 | **RED 阶段完成** | 有压力场景测试记录（基线行为） | 仅限参考文档型技能 |
| 2 | **违规记录完整** | 记录了 Agent 的具体行为和合理化借口 | 同上 |
| 3 | **GREEN 阶段完成** | 用技能重新测试，Agent 合规 | 所有技能必须 |
| 4 | **漏洞修补完成** | REFACTOR 阶段发现的新漏洞已修补 | 规范强制型技能必须 |
| 5 | **合理化对照表** | 有完整的借口对照表（见 writing-rules R10） | 规范强制型技能必须 |
| 6 | **红旗警告列表** | 有明确的"停下来重新开始"检查清单 | 规范强制型技能建议有 |

### 豁免说明模板

如果确实无法进行完整 TDD 测试，必须在技能中注明：

```markdown
---
tdd_status: exempted
exemption_reason: "参考文档型技能，已通过检索场景和应用场景测试"
alternative_validation: "已完成 gap testing 和 retrieval testing"
---
```

### 压力场景类型参考

| 压力类型 | 适用技能 | 设计要点 |
|---------|---------|---------|
| **时间压力** | 规范强制型（TDD、调试） | "每分钟损失 $X" |
| **权威压力** | 流程强制型（代码审查） | "经理/用户要求..." |
| **疲劳/厌倦** | 重复性任务 | "这是第 N 个类似的..." |
| **过度自信** | 复杂技术型 | "这很明显不需要..." |

> 📖 **详细方法论**: [./writing-rules.md](./writing-rules.md) — R8: TDD 驱动技能创建

## 技能类型 × 测试策略矩阵（v6.0 新增）

不同类型的技能需要不同的测试方法。在创建技能时，先判定类型，再选择对应的测试策略。

### 类型定义

| 类型 | 定义 | 示例 |
|------|------|------|
| **规范强制型** | 强制 Agent 遵守规则或流程 | TDD、代码审查、先设计后编码 |
| **技术方法型** | 教授具体的技术或方法 | 条件等待、根因追踪、防御编程 |
| **思维模式型** | 传授思维模型或设计理念 | 简化复杂性、信息隐藏、YAGNI |
| **参考文档型** | API 文档、命令参考、语法指南 | Git 命令参考、库 API 文档 |

### 测试策略矩阵

| 技能类型 | RED 阶段测试方法 | GREEN 阶段验证 | REFACTOR 重点 | 成功标准 |
|---------|-----------------|---------------|--------------|---------|
| **规范强制型** | 学术提问 + 压力场景 + 多重压力组合 | Agent 在最大压力下仍遵守规则 | 发现新合理化方式并禁止 | Agent 无法找到任何绕过规则的方式 |
| **技术方法型** | 应用场景 + 边界情况 + 信息缺失测试 | Agent 能正确应用技术到新场景 | 处理未覆盖的边缘情况 | Agent 在各种边界情况下都能正确使用技术 |
| **思维模式型** | 识别场景 + 反例测试 | Agent 知道何时/如何/何时不应用模式 | 识别误用情况并添加警告 | Agent 能正确判断模式的适用性 |
| **参考文档型** | 检索场景 + 应用场景 + 空白测试 | Agent 能找到并正确使用信息 | 补充常见用例的缺失信息 | Agent 的信息检索准确率 ≥90% |

### 测试深度要求

| 分析模式 | 核心模块覆盖率 | 次要模块覆盖率 | 适用场景 |
|---------|-------------|-------------|---------|
| 快速测试 | ≥30% | ≥10% | Type 1 简单技能 |
| 标准测试（推荐） | ≥60% | ≥30% | Type 2-3 标准技能 |
| 深度测试 | ≥90% | ≥60% | Type 4 复杂技能 / 规范强制型 |
