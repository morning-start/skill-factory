# 🔗 Skill Factory Assembler — 技能整合器 v2.0

> **定位**: 多技能合并与拆分的架构重组器
> **架构**: 单文件自含型子技能（可独立通过 `/assembler` 触发）
> **核心能力**: 合并模式（序列/并行/嵌套）+ 拆分维度（功能/场景/角色）

---

## 🎯 职责范围

| ✅ 负责 | ❌ 不负责 |
|---------|----------|
| 合并多个技能为一个 | 从零创建技能 → `/creator` |
| 拆分复杂技能为多个 | 优化单个技能 → `/processor` |
| 技能架构重组与调整 | 发布/版本管理 → `/publisher` |
| 子技能合并/拆分决策 | 质量审计 → `/processor` |

## 模式一：合并（Merge）

### 第一步：分析源技能（10 min）

#### 1.1 收集源技能信息

```markdown
## 源技能清单

### 技能 A: {name}
- 路径: {path}
- 类型: Type N
- 功能: {一句话描述}
- 行数: {lines}
- 依赖: {dependencies}
- 子技能数: {N}

### 技能 B: {name}
...

### 技能 C: {name} (如有更多)
...
```

#### 1.2 兼容性分析

```markdown
## 合并可行性检查

### ✅ 适合合并的信号
- [ ] 功能高度相关（同一领域）
- [ ] 经常被一起使用
- [ ] 有共享的 references/
- [ ] 用户经常混淆两者的职责
- [ ] 单独使用时功能不完整

### ❌ 不适合合并的信号
- [ ] 功能领域完全不同
- [ ] 各自有独立的用户群体
- [ ] 合并后会超过 Type 4 复杂度上限
- [ ] 生命周期节奏不一致（一个频繁更新，一个稳定）
- [ ] 强制合并会违反单一职责原则

### 兼容性评分
总分: __ /10
(≥7 分建议合并, <5 分不建议)
```

#### 1.3 冲突检测

```markdown
## 潜在冲突识别

### Front Matter 冲突
| 字段 | 技能A | 技能B | 冲突? | 解决方案 |
|------|-------|-------|-------|---------|
| name | - | - | - | 取新名称 |
| version | - | - | - | 取最低版作为基础 |
| description | - | - | - | 重新编写 |
| tags | - | - | - | 合并去重 |

### 内容冲突
| 章节 | 技能A | 技能B | 冲突程度 | 解决方案 |
|------|-------|-------|---------|---------|
| 目标定义 | - | - | 高/中/低 | {方案} |
| 操作步骤 | - | - | 高/中/低 | {方案} |

### 引用冲突
- 相同文件名在不同路径?
- 外部链接重复?
- 循环引用风险?
```

---

### 第二步：选择合并模式（5 min）

#### 三种合并模式

| 模式 | 适用场景 | 特征 | 示例 |
|------|---------|------|------|
| **序列合并 (Sequential)** | 有先后依赖关系 | A→B→C 顺序执行 | 验证→测试→部署 |
| **并行合并 (Parallel)** | 独立功能可同时存在 | A/B/C 并列子技能 | 格式化+Lint+类型检查 |
| **嵌套合并 (Nested)** | 一个是另一个的子流程 | B 嵌入 A 作为步骤 | 部署中的健康检查 |

#### 模式选择决策树

```
源技能间的关系？
│
├── 有明确的先后顺序
│   └── → 序列合并 (Sequential)
│
├── 功能独立，无依赖
│   └── → 并行合并 (Parallel)
│
├── 一个是另一个的子流程/组成部分
│   └── → 嵌套合并 (Nested)
│
└── 关系复杂，多种并存
    └── → 组合模式（主模式 + 局部辅助模式）
```

### 第三步：执行合并（20-40 min）

#### 3.1 序列合并操作

**场景**: 将"代码审查"和"代码提交"两个技能合并

```
合并前:
  code-reviewer/SKILL.md     (审查逻辑)
  git-committer/SKILL.md     (提交逻辑)

合并后:
  code-pipeline/
  ├── SKILL.md               (协调器: 先审查后提交)
  ├── skills/
  │   ├── reviewer/SKILL.md  (原 code-reviewer)
  │   └── committer/SKILL.md (原 git-committer)
  └── references/
      └── shared-standards.md (共享标准)
```

**操作步骤**：

1. **创建新的根目录**
2. **将各源技能移入 skills/ 作为子技能**
3. **编写协调器 SKILL.md**（路由 + 顺序说明）
4. **提取公共内容到 references/**
5. **更新所有内部引用路径**
6. **统一 Front Matter 格式**

#### 3.2 并行合并操作

**场景**: 将"格式化"、"Lint"、"类型检查"三个技能合并

```
合并前:
  formatter/SKILL.md
  linter/SKILL.md
  type-checker/SKILL.md

合并后:
  code-quality/
  ├── SKILL.md               (协调器: 可选运行任意组合)
  └── skills/
      ├── formatter/SKILL.md
      ├── linter/SKILL.md
      └── type-checker/SKILL.md
```

**关键设计**：
- 协调器提供"全量运行"和"选择性运行"两种模式
- 每个子技能完全独立可用
- 可灵活组合：只运行 formatter+linter，或三者全运行

#### 3.3 嵌套合并操作

**场景**: 将"健康检查"嵌入"部署流程"

```
合并前:
  deployer/SKILL.md          (部署逻辑)
  health-checker/SKILL.md    (健康检查逻辑)

合并后:
  deployer/
  ├── SKILL.md               (部署协调器)
  ├── skills/
  │   └── health-checker/    (部署后的自动检查步骤)
  └── references/
      └── deployment-config.md
```

**关键设计**：
- 被嵌套的技能成为父技能的一个 Phase/Step
- 保留独立调用能力（可通过 `/health-checker` 单独使用）
- 明确数据流：上一步输出 → 下一步输入

---

### 第四步：验证合并结果（15 min）

```markdown
## 合并后验证清单

### 结构验证
- [ ] 目录深度 ≤ 3 层
- [ ] 协调器为轻量路由（<200 行）
- [ ] 每个子技能独立可用（有自己的 description）
- [ ] references/ 仅放共享内容

### 功能验证
- [ ] 所有原有功能保留
- [ ] 合并后的流程端到端可执行
- [ ] 无循环引用
- [ ] 内部链接全部有效

### 质量验证
- [ ] 运行审计脚本，每个子技能 ≥60 分
- [ ] 协调器 CSO description 有效
- [ ] 无冗余内容（去重完成）
- [ ] 统一的命名和格式风格

### 用户验证
- [ ] 合并后的路由清晰易懂
- [ ] 复合场景处理合理
- [ ] 歧义情况有明确指引
```

## 模式二：拆分（Split）

### 第一步：分析目标技能（10 min）

#### 1.1 拆分必要性评估

```markdown
## 拆分可行性检查

### ✅ 需要拆分的信号
- [ ] 单一 SKILL.md >500 行且无法再精简
- [ ] 包含多个独立的功能模块
- [ ] 不同用户群体只需要其中部分功能
- [ ] 维护时经常只改其中一部分
- [ ] 类型判定与实际结构不匹配（如声明 Type 1 但实际像 Type 4）

### ❌ 不需要拆分的信号
- [ ] 总行数 <300 且结构清晰
- [ ] 功能高度内聚，拆开后反而增加复杂度
- [ ] 各部分强耦合，拆分成本过高
- [ ] 当前已经是合理的 Type 2/4 结构
- [ ] 用户反馈良好，无"太复杂"抱怨

### 拆分收益评估
预期收益:
- 可维护性提升: {高/中/低}
- 可读性提升: {高/中/低}
- 复用性提升: {高/中/低}
- 加载效率提升: {高/中/低}
- 总体评分: __ /10 (≥7 建议拆分)
```

#### 1.2 当前结构分析

```markdown
## 当前技能解剖

### 功能模块识别
| 模块 | 行数范围 | 功能描述 | 独立性 | 耦合度 |
|------|---------|---------|--------|--------|
| 模块A | Lx-Ly | {描述} | 高/中/低 | 高/中/低 |
| 模块B | Lx-Ly | {描述} | 高/中/低 | 高/中/低 |
| 模块C | Lx-Ly | {description} | 高/中/低 | 高/中/低 |

### 依赖关系图
模块A ←→ 模块B (强/弱/无依赖)
模块A ←→ 模块C (强/弱/无依赖)
模块B ←→ 模块C (强/弱/无依赖)

### 共享资源识别
- 共享 references/: {列表}
- 共享配置: {列表}
- 共享工具/脚本: {列表}
```

### 第二步：选择拆分维度（5 min）

#### 三种拆分维度

| 维度 | 适用场景 | 特征 | 示例 |
|------|---------|------|------|
| **按功能拆分 (Function)** | 不同功能模块清晰可辨 | 每个子技能负责一个功能域 | CRUD→Create/Read/Update/Delete |
| **按场景拆分 (Scene)** | 同一功能在不同场景下差异大 | 按使用场景划分子技能 | 部署→开发/测试/生产环境 |
| **按角色拆分 (Role)** | 不同角色关注点不同 | 按目标用户划分子技能 | CI/CD→开发者/DevOps/管理员 |

#### 维度选择指南

```
主要区分因素是什么？

├── 功能边界清晰
│   └── → 按功能拆分 (Function)
│
├── 使用场景差异大
│   └── → 按场景拆分 (Scene)
│
├── 用户角色不同
│   └── → 按角色拆分 (Role)
│
└── 多种因素混合
    └── → 主维度 + 辅助维度组合
```

#### 维度对比示例

**原始技能**: DevOps Toolkit (600 行)

| 拆分维度 | 结果 | 优缺点 |
|---------|------|--------|
| **按功能** | deployer / monitor / scaler | ✅ 清晰; ❌ 可能跨场景重复 |
| **按场景** | dev-env / staging / prod | ✅ 场景完整; ❌ 功能分散 |
| **按角色** | developer / ops / admin | ✅ 角色聚焦; ❌ 可能粒度过细 |

**推荐**: 对于 DevOps Toolkit，**按功能拆分** 最自然

### 第三步：执行拆分（20-30 min）

#### 3.1 拆分操作通用流程

```
① 定义拆分方案
   ↓ 确定几个子技能及其边界
② 创建新目录结构
   ↓ 按目标类型(Type 2/4)搭建骨架
③ 分割内容
   ↓ 将原文分配到各子技能
④ 提取共享内容
   ↓ 公共部分→全局references/, 半公共→各子技能refs
⑤ 编写/更新协调器
   ↓ 如需协调器则编写路由逻辑
⑥ 更新所有引用
   ↓ 修复内部/外部链接
⑦ 验证完整性
   ↓ 功能+质量双重验证
```

#### 3.2 按功能拆分示例

**拆分前**:

```
devops-toolkit/
└── SKILL.md (600行, 包含: 部署/监控/扩缩容)
```

**拆分后**:

```
devops-toolkit/
├── SKILL.md                      (~150行, 协调器)
├── references/
│   └── common-config.md          (共享配置)
└── skills/
    ├── deployer/
    │   ├── SKILL.md              (~180行, 部署专用)
    │   └── references/
    │       └── deployment-strategies.md
    ├── monitor/
    │   ├── SKILL.md              (~150行, 监控专用)
    │   └── references/
    │       └── metrics-guide.md
    └── scaler/
        ├── SKILL.md              (~120行, 扩缩容专用)
        └── references/
            └── scaling-policies.md
```

**关键操作**：

1. **识别功能边界**
   - 部署: 行 50-250
   - 监控: 行 251-420
   - 扩缩容: 行 421-600

2. **提取共享内容**
   - 公共配置参数 → `references/common-config.md`
   - 各功能独有策略 → 各子技能 `references/`

3. **编写协调器**
   - 路由表: "部署"→deployer, "监控"→monitor, "扩缩"→scaler
   - 复合场景: "部署并监控"→deployer→monitor

#### 3.3 按场景拆分示例

**适用**: 同一功能在不同环境下行为差异大

```
testing-toolkit/
├── SKILL.md                      (协调器)
└── skills/
    ├── unit-tester/              (单元测试场景)
    ├── integration-tester/       (集成测试场景)
    └── e2e-tester/               (端到端测试场景)
```

#### 3.4 按角色拆分示例

**适用**: 不同角色需要不同的信息密度和操作权限

```
project-manager/
├── SKILL.md                      (协调器)
└── skills/
    ├── developer-view/           (开发者视角)
    ├── pm-view/                  (项目经理视角)
    └── stakeholder-view/         (利益相关者视角)
```

## 📊 合并 vs 拆分对比

| 维度 | 合并 | 拆分 |
|------|------|------|
| **触发词** | 合并/整合/组合/统一 | 拆分/分离/解耦/精简 |
| **输入** | 2-4 个源技能 | 1 个目标技能 |
| **输出** | 1 个整合技能 | 2-4 个子技能 |
| **核心挑战** | 解决冲突 | 保持完整性 |
| **典型耗时** | 30-50 min | 30-45 min |
| **风险等级** | 中（可能引入耦合） | 中（可能丢失功能） |

---

## ⚠️ 注意事项

1. **合并不等于简单拼接**: 要解决冲突、重新组织、统一风格
2. **拆分不等于机械切割**: 要保证每个子技能的独立性和完整性
3. **先评估再行动**: 不是所有技能都适合合并或拆分
4. **保持向后兼容**: 如果技能已被使用，拆分时要保留原入口
5. **文档同步更新**: 合并/拆分后要更新所有相关文档和路由
6. **渐进式重组**: 大型重构可分多步完成，不必一次到位
