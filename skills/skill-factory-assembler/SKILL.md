# 技能整合器

> **任务目标**: 帮助用户合并多个独立技能或拆分过于庞大的技能，确保操作后所有技能均满足三层架构铁律。

处理 **多技能合并** 和 **复杂技能拆分** 的场景。

---

## 触发条件

| 用户说 | 场景 |
|--------|------|
| "把 A 和 B 合并成一个技能" | 多技能合并 |
| "这个技能太大了，拆开来" | 复杂技能拆分 |
| "这些功能能整合在一起吗" | 整合评估 |
| "把XX功能从主技能里分出来" | 功能拆分 |

---

## 操作步骤

### 一、合并（Merge）

### 合并流程

```
分析职责 → 选择模式 → 规划层级结构 → 合并实现 → 验证
```

### 三种合并模式

#### 1. 序列模式（A → B → C）

有前后依赖的操作，上一步的输出是下一步的输入。

```yaml
# 合并后的前言区
dependency:
  mode: sequential
  steps:
    - skill-A  # 第一步
    - skill-B  # 第二步（依赖 A 的输出）
    - skill-C  # 第三步（依赖 B 的输出）
```

**适用场景**：流水线操作、有固定顺序的多步骤流程。

**示例**：代码检查 → 代码修复 → 代码格式化

#### 2. 并行模式（A / B / C）

无依赖关系的独立操作，可以同时执行。

```yaml
dependency:
  mode: parallel
  tasks:
    - skill-A  # 可独立执行
    - skill-B  # 可独立执行
    - skill-C  # 可独立执行
```

**适用场景**：多维度分析、多格式导出、独立功能集。

**示例**：性能分析 + 安全审计 + 代码规范检查

#### 3. 嵌套模式（B 嵌入 A）

一个技能作为另一个技能的特定步骤或子流程。

```yaml
dependency:
  mode: nested
  host: skill-A
  guests:
    - skill-B  # 在 A 的步骤3 中调用
```

**适用场景**：主流程中某步骤需要子流程。

**示例**：部署流程中的"健康检查"步骤

### 合并规范

| 规则 | 说明 |
|------|------|
| 层级约束 | 合并后总层级 ≤3 |
| 命名统一 | 目录和文件名统一为 kebab-case |
| 子技能父级 | 合并后的子技能 parent 指向新父技能 |
| 整合验证 | 合并后对照 skill-standards.md 完整验证 |

## 二、拆分（Split）

### 拆分流程

```
评估复杂度 → 选择维度 → 设计边界 → 拆分实现 → 验证（≤3层）
```

### 三种拆分维度

#### 1. 按功能拆分

按独立的功能域拆为独立技能。

```
原技能 "user-management" (600行)
         ↓ 按功能拆分
user-auth/              ← 认证功能（300行）
user-profile/           ← 资料管理（200行）
user-permissions/       ← 权限控制（250行）
```

#### 2. 按场景拆分

按使用场景或环境拆分。

```
原技能 "deployment" (500行)
         ↓ 按场景拆分
deploy-dev/             ← 开发环境部署
deploy-prod/            ← 生产环境部署
deploy-rollback/        ← 回滚操作
```

#### 3. 按角色拆分

按使用者角色拆分。

```
原技能 "dashboard" (550行)
         ↓ 按角色拆分
dashboard-admin/        ← 管理员视图
dashboard-user/         ← 普通用户视图
```

**非编程领域示例**：

```
原技能 "design-system" (700行)
         ↓ 按角色拆分
design-tokens/           ← 设计变量（200行）
design-components/       ← 组件库（300行）
design-guidelines/       ← 设计规范（250行）

原技能 "api-docs" (550行)
         ↓ 按场景拆分
api-docs-getting-started/← 入门教程（150行）
api-docs-reference/      ← API 参考（350行）
api-docs-migration/      ← 迁移指南（180行）
```

### 拆分铁律

- 每个拆分后的技能 **≤3 层**
- 拆分后的技能 **独立可运行**（不依赖原技能）
- 原技能添加迁移指引（指向新技能族）
- 若拆分后仍超层，继续拆分

---

## 三、验证清单

合并或拆分后必须验证：

| # | 检查项 | 标准 |
|---|--------|------|
| 1 | 层级合规 | 所有技能 ≤3 层 |
| 2 | 命名规范 | kebab-case，无冲突 |
| 3 | 前言区完整 | 每个子技能有独立前言区 |
| 4 | parent 正确 | 指向合并/拆分后的新父技能 |
| 5 | 链接有效 | 无死链，references 路径正确 |
| 6 | 迁移指引 | 原位置有废弃标记和迁移说明 |

---

## ⚠️ 注意事项

- **合并前先评估**：不是所有技能都适合合并——如果两个技能的核心能力差异太大，强行合并会导致职责混乱
- **拆分后必须独立**：每个拆分出的子技能都应该是可以独立运行和理解的完整技能，不能只是"原技能的一部分"
- **层级是硬约束**：合并或拆分后任一子技能超过 3 层即为失败，必须重新设计方案
- **迁移指引不能省**：原技能废弃后必须有清晰的迁移路径，不能让依赖方不知所措

## 快速决策表

| 场景 | 模式 | 关键动作 |
|------|------|---------|
| 有依赖的操作合并 | 序列 | 定义步骤顺序和依赖关系 |
| 独立的操作合并 | 并行 | 确保无共享状态 |
| 嵌入子流程 | 嵌套 | 明确调用点和参数传递 |
| 超大单体拆分 | 功能 | 识别独立功能域 |
| 多环境拆分 | 场景 | 提取环境差异 |
| 多权限拆分 | 角色 | 提取权限差异 |
