# EvoSkill 多候选评估方法论

> **来源**: 中文社区 jishuzhan.net EvoSkill 模式 + skill-factory assembler 扩展
> **何时读取**: 合并/拆分操作存在多个可行策略时，需生成多候选方案并自动选优

---

## 核心流程

```
Creator Agent → Evaluator Agent → Refiner Agent
     ↓              ↓               ↓
  生成候选      结构化评分       迭代优化
 (2-3个方案)   (6维评分卡)     (基于反馈)
```

### 三角色职责

| 角色 | 职责 | 输入 | 输出 |
|------|------|------|------|
| **Creator** | 基于不同策略生成多个完整候选方案 | 源技能/目标技能 + 策略列表 | 2-3 个完整合并/拆分方案 |
| **Evaluator** | 对每个候选运行结构化评分 | 所有候选方案 | 6 维评分卡 + 排名 |
| **Refiner** | 根据评估反馈优化最优方案 | 最优方案 + 扣分项 | 最终优化版本 |

---

## 触发条件

> ⚠️ **何时使用 EvoSkill 模式**

```markdown
满足以下任一条件时启用:
- [ ] 存在 2+ 种合理的合并/拆分策略（无法直观判断哪种更优）
- [ ] 技能规模较大（>400行），策略选择对结果影响显著
- [ ] 用户要求"给出最佳方案"或"比较不同方法"
- [ ] 首次操作该类技能组合，缺乏先例可循

不满足时使用标准单方案流程即可。
```

## 适用于 assembler 的场景

### 场景一：多策略合并（Merge）

当有 N 个待合并技能时，不同合并策略产生不同结果：

| 策略 | 方法 | 优点 | 缺点 | 适用 |
|------|------|------|------|------|
| **顺序融合** | A→B→C 依次合并 | 保持逻辑顺序 | 可能丢失并行性 | 有依赖链的技能 |
| **并行汇聚** | 同时分析所有技能再统一 | 全局视角最优 | 复杂度高 | 无明显依赖 |
| **核心提取** | 只合并核心章节，references 保留独立 | 最小变更原则 | 可能有冗余 | 大型技能轻量合并 |

**示例**: 合并 `code-reviewer` + `git-committer` + `test-runner`

```markdown
候选 A (顺序融合):
  code-pipeline/
  ├── SKILL.md          (协调器: review→test→commit)
  └── skills/
      ├── reviewer/
      ├── tester/
      └── committer/

候选 B (并行汇聚):
  code-quality/
  ├── SKILL.md          (协调器: 可选任意组合)
  └── skills/
      ├── reviewer/
      ├── tester/
      └── committer/

候选 C (核心提取):
  code-workflow/
  ├── SKILL.md          (轻量协调器)
  ├── core-skills.md    (合并后的核心流程)
  └── references/
      ├── review-rules.md    (独立保留)
      ├── test-strategies.md (独立保留)
      └── commit-guide.md    (独立保留)
```

### 场景二：多维拆分（Split）

按不同维度拆分产生不同方案：

| 维度 | 拆分方式 | 评估指标 |
|------|---------|---------|
| **功能拆分** | 按功能点切分 | 每个子技能内聚度 |
| **场景拆分** | 按使用场景切分 | 触发率分布均匀度 |
| **角色拆分** | 按使用者角色切分 | 用户群体匹配度 |

**示例**: 拆分 `devops-toolkit` (600行, 含部署/监控/扩缩容)

```markdown
候选 A (功能拆分):
devops-toolkit/
├── SKILL.md
└── skills/
    ├── deployer/        (部署专用)
    ├── monitor/         (监控专用)
    └── scaler/          (扩缩容专用)

候选 B (场景拆分):
devops-toolkit/
├── SKILL.md
└── skills/
    ├── daily-ops/       (日常运维场景)
    ├── incident/        (故障处理场景)
    └── scaling/         (弹性伸缩场景)

候选 C (角色拆分):
devops-toolkit/
├── SKILL.md
└── skills/
    ├── dev-view/        (开发者视角)
    ├── ops-view/        (运维视角)
    └── admin-view/      (管理员视角)
```

## 6 维评分卡

### 评分维度定义

| 维度 | 权重 | 评估方法 | 满分 | 关键问题 |
|------|------|---------|------|---------|
| **功能完整性** | 25% | 原始功能是否全部保留？ | 25 | 有无遗漏？有无阉割？ |
| **结构合理性** | 20% | 层级/行数/分类是否合理？ | 20 | 是否符合 Type 判定？深度≤3？ |
| **CSO 质量** | 20% | description 是否清晰可触发？ | 20 | 能否被 Agent 准确识别调用？ |
| **冗余度** | 15% | 有无重复内容？ | 15 | 去重是否彻底？有无信息冗余？ |
| **可维护性** | 10% | 后续是否易于修改？ | 10 | 改一处是否会波及多处？ |
| **兼容性** | 10% | 与现有生态是否兼容？ | 10 | 是否破坏现有调用链？ |

### 评分标准参考

#### 功能完整性 (25分)

| 分数段 | 标准 |
|--------|------|
| 23-25 | 所有原始功能100%保留，无任何遗漏 |
| 18-22 | 核心功能完整，次要功能有轻微精简 |
| 13-17 | 主要功能保留，部分边缘功能被裁剪 |
| 0-12 | 存在明显功能缺失或逻辑断裂 |

#### 结构合理性 (20分)

| 分数段 | 标准 |
|--------|------|
| 18-20 | 完全符合 skill-factory Type 规范，层级清晰 |
| 14-17 | 基本合规，个别地方可优化 |
| 10-13 | 结构可用但存在不规范之处 |
| 0-9 | 明显违反设计原则（如过深嵌套、混乱分类） |

#### CSO 质量 (20分)

| 分数段 | 标准 |
|--------|------|
| 18-20 | description 精准、触发词丰富、歧义少 |
| 14-17 | description 可用，触发词覆盖主要场景 |
| 10-13 | description 存在模糊处，可能误触发 |
| 0-9 | description 不清晰或缺失关键触发词 |

#### 冗余度 (15分 - 越低越好，此处为扣分制)

| 分数段 | 标准 |
|--------|------|
| 13-15 | 去重彻底，无任何重复内容 |
| 10-12 | 存在少量可接受的重复（如跨章节引用） |
| 6-9 | 有明显重复但未影响使用 |
| 0-5 | 大量冗余，严重影响阅读和维护 |

#### 可维护性 (10分)

| 分数段 | 标准 |
|--------|------|
| 9-10 | 模块化好，修改影响范围小且可控 |
| 7-8 | 基本模块化，少量耦合 |
| 5-6 | 存在一定耦合，修改需注意连锁反应 |
| 0-4 | 高度耦合，牵一发而动全身 |

#### 兼容性 (10分)

| 分数段 | 标准 |
|--------|------|
| 9-10 | 完全向后兼容，现有调用无需改动 |
| 7-8 | 兼容主流场景，少数边缘情况需适配 |
| 5-6 | 需要迁移指南，存在 breaking change |
| 0-4 | 破坏性变更，大量现有用法失效 |

## 评估执行步骤

### Phase 1: Creator — 生成候选方案

```markdown
## 输入分析
- 操作类型: Merge / Split
- 涉及技能: {列表}
- 可行策略: {识别出的策略列表}

## 候选方案生成
基于每种策略生成一个完整的、可直接使用的方案。

要求:
1. 每个候选必须是完整的（含目录结构、SKILL.md 内容）
2. 候选间应有明显差异（不是微小变体）
3. 数量控制在 2-3 个（过多会增加评估成本）
```

### Phase 2: Evaluator — 结构化评分

对每个候选运行 6 维评分卡，填写对比表。

### Phase 3: Refiner — 选优与优化

```markdown
## 决策过程
1. 计算加权总分
2. 选择最高分方案作为基础
3. 分析其他方案的优点（是否有值得采纳的设计？）
4. 将其他方案的优势整合到最优方案中
5. 输出最终优化版本
```

## 最佳实践

### ✅ 推荐做法

1. **策略先行**: 先列出所有可行策略，再决定是否需要多候选评估
2. **差异最大化**: 确保候选方案之间有本质差异，而非表面变化
3. **评分客观化**: 尽可能量化评分依据，避免主观偏好
4. **日志留存**: 始终记录评估日志，便于后续审查和学习
5. **成本控制**: 只有在策略选择确实困难时才启动 EvoSkill 流程

### ❌ 常见误区

| 误区 | 说明 | 正确做法 |
|------|------|---------|
| 过度评估 | 简单场景也生成多候选 | 先判断是否真的需要多候选 |
| 候选同质化 | 多个候选只是微调差异 | 确保采用不同核心策略 |
| 忽略 Refiner | 选出最高分就直接输出 | 整合其他方案优点进行优化 |
| 评分主观化 | 凭感觉打分 | 严格按评分标准逐项评估 |
| 缺少日志 | 评估完就丢弃中间结果 | 始终保存完整评估记录 |

## 与标准流程的关系

```
标准 Assembler 流程:
  分析 → 选模式 → 执行 → 验证

EvoSkill 增强流程:
  分析 → 识别多策略? 
           ├─ 否 → 标准流程（单方案）
           └─ 是 → EvoSkill 流程：
                    ├─ Creator: 生成 2-3 候选
                    ├─ Evaluator: 6维评分
                    ├─ Refiner: 选优+优化
                    └─ 验证
```
