## 第一步：分析目标技能

### 拆分必要性评估

| ✅ 需要拆分的信号 | ❌ 不需要拆分的信号 |
|------------------|-------------------|
| >500 行且无法再精简 | <300 且结构清晰 |
| 多个独立的功能模块 | 功能高度内聚 |
| 不同用户只需部分功能 | 各部分强耦合 |
| 维护时只改其中一部分 | 已是合理 Type 2/4 结构 |
| 类型判定与实际不匹配 | 用户反馈良好 |

## 第三步：执行拆分

### 通用流程

```
① 定义拆分方案 → ② 创建目录骨架 → ③ 分割内容
       ↓
④ 提取共享内容 → ⑤ 编写协调器 → ⑥ 更新引用 → ⑦ 验证
```

### 3.1 按功能拆分 (Function)

**适用**: 不同功能模块清晰可辨

**原始**: DevOps Toolkit (600行, 含部署/监控/扩缩容)

```
拆分后:
devops-toolkit/
├── SKILL.md                      (~150行, 协调器)
├── references/common-config.md
└── skills/
    ├── deployer/                 (部署专用 ~180行)
    ├── monitor/                  (监控专用 ~150行)
    └── scaler/                   (扩缩容专用 ~120行)
```

**关键操作**:
1. 识别功能边界（按行范围划分）
2. 提取共享配置到 `references/common-config.md`
3. 各功能独有策略放各子技能 `references/`
4. 编写路由: "部署"→deployer, "监控"→monitor

### 3.2 按场景拆分 (Scene)

**适用**: 同一功能在不同环境下行为差异大

```
testing-toolkit/
├── SKILL.md (协调器)
└── skills/
    ├── unit-tester/              (单元测试场景)
    ├── integration-tester/       (集成测试场景)
    └── e2e-tester/               (端到端测试场景)
```

### 3.3 按角色拆分 (Role)

**适用**: 不同角色需要不同信息密度和权限

```
project-manager/
├── SKILL.md (协调器)
└── skills/
    ├── developer-view/           (开发者视角)
    ├── pm-view/                  (项目经理视角)
    └── stakeholder-view/         (利益相关者视角)
```
