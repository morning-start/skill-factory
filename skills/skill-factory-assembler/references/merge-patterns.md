# 合并模式详解 — 序列/并行/嵌套

> **来源**: [../SKILL.md](../SKILL.md) → 模式一：合并
> **何时读取**: 执行技能合并操作时

---

## 第一步：分析源技能

### 源技能信息收集

```markdown
## 源技能清单

### 技能 A: {name}
- 路径: {path} | 类型: Type N | 行数: {lines}
- 功能: {一句话描述} | 子技能数: {N}

### 技能 B: {name}
...
```

### 兼容性分析

| ✅ 适合合并的信号 | ❌ 不适合合并的信号 |
|------------------|-------------------|
| 功能高度相关（同一领域） | 功能领域完全不同 |
| 经常被一起使用 | 各自有独立用户群体 |
| 有共享的 references/ | 合并后超 Type 4 上限 |
| 用户经常混淆两者的职责 | 生命周期节奏不一致 |
| 单独使用时功能不完整 | 强制合并违反单一职责 |

**兼容性评分**: ___ /10 （≥7 建议合并, <5 不建议）

### 冲突检测

**Front Matter 冲突**:

| 字段 | 技能A | 技能B | 解决方案 |
|------|-------|-------|---------|
| name | - | - | 取新名称 |
| version | - | - | 取最低版作为基础 |
| description | - | - | 重新编写 |
| tags | - | - | 合并去重 |

**内容冲突**: 目标定义 / 操作步骤 → 评估冲突程度(高/中/低)

**引用冲突**: 相同文件名? 外部链接重复? 循环引用?

## 第三步：执行合并

### 3.1 序列合并 (Sequential)

**适用**: 有先后依赖关系 → A→B→C 顺序执行

**场景**: 将"代码审查"和"代码提交"合并为"代码流水线"

```
合并前:
  code-reviewer/SKILL.md     git-committer/SKILL.md

合并后:
  code-pipeline/
  ├── SKILL.md               (协调器: 先审查后提交)
  ├── skills/
  │   ├── reviewer/SKILL.md
  │   └── committer/SKILL.md
  └── references/
      └── shared-standards.md
```

**操作步骤**:
1. 创建新的根目录
2. 将各源技能移入 `skills/` 作为子技能
3. 编写协调器 SKILL.md（路由 + 顺序说明）
4. 提取公共内容到 `references/`
5. 更新所有内部引用路径
6. 统一 Front Matter 格式

### 3.2 并行合并 (Parallel)

**适用**: 独立功能可同时存在 → A/B/C 并列子技能

**场景**: 将"格式化+Lint+类型检查"合并为"代码质量"

```
合并后:
  code-quality/
  ├── SKILL.md               (协调器: 可选运行任意组合)
  └── skills/
      ├── formatter/SKILL.md
      ├── linter/SKILL.md
      └── type-checker/SKILL.md
```

**关键设计**:
- 协调器提供"全量运行"和"选择性运行"两种模式
- 每个子技能完全独立可用
- 可灵活组合

### 3.3 嵌套合并 (Nested)

**适用**: 一个是另一个的子流程 → B 嵌入 A 作为步骤

**场景**: 将"健康检查"嵌入"部署流程"

```
合并后:
  deployer/
  ├── SKILL.md               (部署协调器)
  ├── skills/
  │   └── health-checker/    (部署后的自动检查)
  └── references/
      └── deployment-config.md
```

**关键设计**:
- 被嵌套的技能成为父技能的一个 Phase/Step
- 保留独立调用能力（可通过 `/health-checker` 单独使用）
- 明确数据流：上一步输出 → 下一步输入

## 第四步：验证合并结果

```markdown
### 结构验证
- [ ] 目录深度 ≤ 3 层
- [ ] 协调器 <200 行
- [ ] 每个子技能独立可用（有自己的 description）
- [ ] references/ 仅放共享内容

### 功能验证
- [ ] 所有原有功能保留
- [ ] 端到端可执行
- [ ] 无循环引用
- [ ] 内部链接有效

### 质量验证
- [ ] 运行审计脚本，每个子技能 ≥60 分
- [ ] CSO description 有效
- [ ] 无冗余（去重完成）
- [ ] 统一风格

### 用户验证
- [ ] 路由清晰易懂
- [ ] 复合场景合理
- [ ] 歧义有明确指引
```
