## 评分体系概览

### 总分: 100 分 = 7 个维度

| # | 维度 | 权重 | 核心关注点 | 一票否决? |
|---|------|------|-----------|----------|
| 1 | **Front Matter 完整** | 10 分 | 基础元数据 | 否 |
| 2 | **CSO Description** | **15 分** | **触发质量** | **是** |
| 3 | **TDD 验证** | **15 分** | **测试覆盖** | **是** |
| 4 | **必备章节** | 10 分 | 结构完整性 | 否 |
| 5 | **层级合规** | 10 分 | 架构合规 | **是** |
| 6 | **命名规范** | 5 分 | 命名一致性 | 否 |
| 7 | **链接有效性** | 5 分 | 引用可靠性 | 否 |

### 等级定义

| 分数段 | 等级 | 颜色 | 含义 | 发布建议 |
|--------|------|------|------|---------|
| **90-100** | ⭐⭐⭐ A | 🟢 优秀 | 高质量，可直接发布 | ✅ 立即发布 |
| **75-89** | ⭐⭐ B | 🟢 良好 | 质量良好，有小瑕疵 | ✅ 小修后发布 |
| **60-74** | C | 🟡 合格 | 基本合格，需优化 | ⚠️ 建议优化后发布 |
| **<60** | ❌ D | 🔴 不合格 | 有重大缺陷 | 🚫 必须修复 |

## 各维度评分细则

### 维度 1: Front Matter 完整（10 分）

**检查项**：

| 字段 | 分值 | 通过标准 | 扣分 |
|------|------|---------|------|
| name | 2.5 分 | kebab-case，≤50 字符 | -2.5/缺失或违规 |
| version | 2.5 分 | v{major}.{minor}.{patch} 格式 | -2.5/格式错误 |
| description | 2.5 分 | 非空，有实际内容 | -2.5/空或占位符 |
| tags | 2.5 分 | 数组格式，≥1 个标签 | -2.5/缺失或格式错 |

**满分条件**: 四个字段全部存在且格式正确

**常见扣分点**：
- name 用驼峰或空格: -2.5
- version 缺少或用 "1.0" 而非 "v1.0.0": -2.5
- description 为空或 "...": -2.5
- tags 缺失: -2.5

---

### 维度 2: CSO Description（15 分）⭐ 核心

**检查项**：

| 规则 | 分值 | 通过标准 | 扣分 |
|------|------|---------|------|
| **包含 "Use when"** | 5 分 | description 中含有 "Use when" 短语 | -5/缺失 |
| **无工作流总结** | 5 分 | 不包含 workflow/steps/execute/process 等词 | -5/包含 |
| **长度合理** | 5 分 | 50-1024 字符 | -2/<50 或 -3/>1024 |

**为什么这是核心？**
- description 是 L1 Metadata，始终加载到系统提示
- 写得不好 → 技能永远不会被自动触发
- 总结了工作流 → Agent 直接执行 description 跳过正文

**CSO 编写规范**：

```
✅ 正确格式 (15/15):
  "Use when creating, editing, or optimizing AI Agent skills.
   Triggers on 'create a skill', 'write SKILL.md', or 'optimize skill'.
   Handles type classification and TDD-driven development."
   (198字符, 含Use when, 无工作流词)

❌ 错误格式 (5/15):
  "Skill creation guide with TDD workflow and type classification.
   Helps create high-quality SKILL.md files step by step."
   (138字符, 无Use when, 含workflow/guide)

❌ 错误格式 (7/15):
  "Use when creating skills."
   (25字符, 太短, 覆盖不足)
```

**高级检查项**（不额外加分，但作为质量提示）：

- [ ] 是否包含触发短语同义词？（create/make/build/write）
- [ ] 是否避免了 XML 角括号？
- [ ] 是否以第三人称编写？
- [ ] 是否技术无关（除非技能本身是技术特定的）？

---

### 维度 3: TDD 验证（15 分）⭐ 核心

**检查项**：

| 规则 | 分值 | 通过标准 | 扣分 |
|------|------|---------|------|
| **提及 TDD** | 5 分 | 文档中有 TDD/tdd/stress test/RED/GREEN/REFACTOR | -5/无提及 |
| **测试记录** | 5 分 | 有压力场景/基线测试/test record | -5/无记录 |
| **豁免说明** | 5 分 | 有 waiver/exempt/豁免/skip reason 或完整测试记录 | -5/两者皆无 |

**TDD 豁免条件**（有豁免也得 5 分）：

| 豁免类型 | 可接受的理由 | 需要的字段 |
|---------|------------|-----------|
| Type 1 快速路径 | 单一功能非关键任务 | `tdd: simplified` |
| 参考文档型 | 纯信息检索类 | `tdd: validation-only` |
| 原型/MVP | 实验性质 | 版本号 `0.x-alpha` + `tdd: deferred` |

**豁免声明模板**：

```yaml
meta:
  tdd: simplified
  tdd_waiver_reason: "Type 1 quick path, non-critical task"
  tdd_waiver_date: "2026-05-27"
```

**为什么这是核心？**
- TDD 是 skill-factory 的铁律
- 没有测试就无法证明技能有效
- 但允许合理豁免（需明确声明原因）

---

### 维度 4: 必备章节（10 分）

**检查项**：

| 章节 | 分值 | 通过标准 | 扣分 |
|------|------|---------|------|
| 任务目标/定位 | 2.5 分 | 有 goal/target/目标/定位 章节 | -2.5/缺失 |
| 操作步骤/流程 | 2.5 分 | 有 step/步骤/流程/操作 章节 | -2.5/缺失 |
| 示例/场景 | 2.5 分 | 有 example/示例/场景 | -2.5/缺失 |
| 注意事项 | 2.5 分 | 有 note/caution/注意/Gotchas | -2.5/缺失 |

**进阶检查**（不额外加分，质量提示）：

注意事项的质量分级：

| 等级 | 标准 | 示例 |
|------|------|------|
| ⭐⭐⭐ 优秀 | 有 Gotchas 坑点表 + 错误处理矩阵 + 反模式 | 见 writing-rules R1/R2/R4 |
| ⭐⭐ 良好 | 有具体的坑点或错误分类 | "忘记 `-m` 导致编辑器弹出" |
| ⭐ 合格 | 有注意事项章节（即使泛泛） | "注意操作安全" |
| ❌ 不合格 | 完全没有注意事项章节 | - |

---

### 维度 5: 层级合规（10 分）

**计算规则**：

```
深度计算:
- 从 SKILL.md 所在目录开始计数
- 每深入一层子目录 +1
- references/ scripts/ assets/ 不算层级
- 最大允许深度: 3 层
```

**评分标准**：

| 深度 | 得分 | 说明 |
|------|------|------|
| 0-2 层 | 10/10 | ✅ 完全合规 |
| 3 层 | 10/10 | ✅ 合规（上限） |
| 4 层 | 5/10 | ⚠️ 超限，需警告 |
| ≥5 层 | 0/10 | 🚨 严重超限 |

**示例**：

```
✅ 合规 (10分):
  skill/SKILL.md                              (0层)
  skill/skills/creator/SKILL.md               (2层)
  skill/skills/creator/references/guide.md    (references不算, 仍2层)

⚠️ 警告 (5分):
  skill/skills/a/b/c/SKILL.md                 (4层)

🚨 失败 (0分):
  skill/skills/a/b/c/d/SKILL.md               (5层)
```

**为什么是一票否决？**
- 三层铁律是 skill-factory 的核心设计原则
- 超过 3 层会导致认知负担指数上升
- 影响技能的可维护性和可扩展性

---

### 维度 6: 命名规范（5 分）

**检查项**：

| 对象 | 规范 | 分值 |
|------|------|------|
| SKILL.md 文件 | 固定名称 "SKILL.md" | 通过即得 5 分 |
| 目录名 | kebab-case（小写+连字符） | 违规 -5 分 |

**kebab-case 定义**：

```
✅ 正确:
  skill-factory-creator
  git-commit-helper
  my-awesome-skill

❌ 错误:
  SkillFactoryCreator      (PascalCase)
  skill_factory_creator    (snake_case)
  skillfactorycreator      (无分隔)
  My-Awesome-Skill         (首字母大写)
```

---

### 维度 7: 链接有效性（5 分）

**检查项**：

| 规则 | 分值 | 通过标准 |
|------|------|---------|
| 内部链接有效 | 5 分 | 所有相对路径链接指向存在的文件 |

**检测方法**：
- 扫描所有 `[text](path)` 格式的链接
- 过滤外部链接（http/https/# 开头的不查）
- 检查相对路径（./ 或 ../ 开头）是否存在
- 每个坏链 -1 分，最低 0 分

**常见死链原因**：
- 文件移动后未更新引用路径
- 拼写错误（大小写敏感）
- 引用了已删除的文件
- 路径层级错误（多/少了一层 ../）

## 常见问题与解答

### Q1: 为什么 CSO 和 TDD 权重最高（各 15 分）？

因为这两项决定了技能是否能**有效工作**：
- CSO 决定技能能否被 Agent **激活**
- TDD 决定技能内容是否**经过验证**

其他项更多是"质量保证"，而这两项是"生存基础"。

### Q2: 60 分就能发布吗？

60 分是**最低门槛**，意味着：
- 基础结构完整
- 无阻塞性问题（CSO/TDD/层级都及格）
- 但有明显改进空间

**建议**:
- ≥75 分: 可以发布，持续迭代
- 60-74 分: 建议至少修到 75+ 再发布
- <60 分: 不建议发布，必须修复

### Q3: 审计分数低怎么办？

按以下优先级修复：

1. **P0 (阻塞性)**: CSO / TDD / 层级 — 这些是不合格的根本原因
2. **P1 (重要)**: 必备章节 / Front Matter — 结构完整性
3. **P2 (优化)**: 命名 / 链接 / 内容质量 — 锦上添花

每修复一轮后重新审计，追踪进步。

### Q4: 如何快速提升分数？

**最快提分方法**（按投入产出比排序）：

| 方法 | 预期提升 | 耗时 | 难度 |
|------|---------|------|------|
| 修正 CSO description | +5~10 分 | 10min | 低 |
| 添加 TDD 豁免说明 | +5 分 | 5min | 低 |
| 补充缺失章节 | +2.5~10 分 | 20min | 中 |
| 修复死链 | +1~5 分 | 10min | 低 |
| 修正命名 | +5 分 | 5min | 低 |

## 审计自动化

### 使用 audit.ps1 脚本

位于 `scripts/audit.ps1`，支持：

```powershell
# 单文件审计
.\scripts\audit.ps1 -Path .\target\SKILL.md

# 项目全量审计
.\scripts\audit.ps1 -Project

# 详细模式（显示检查过程）
.\scripts\audit.ps1 -Path .\target\SKILL.md -Verbose
```

### 脚本 vs 手动审计

| 维度 | 脚本审计 | 手动审计 |
|------|---------|---------|
| **速度** | 2 秒 | 15-30 分钟 |
| **客观性** | 100% 客观 | 受主观影响 |
| **覆盖面** | 7 个固定维度 | 可自定义扩展 |
| **深度** | 格式/规则检查 | 内容质量评估 |
| **建议** | 无 | 可生成改进方案 |

**推荐做法**: 脚本初筛 → 手动深入 → 结合使用
