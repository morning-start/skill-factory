# 加工策略详解 — 4 种策略完整指南

> **来源**: [../SKILL.md](../SKILL.md) → 加工模式 → 策略选择
> **版本**: v2.0.0
> **用途**: 根据技能当前状态选择合适的加工策略，并提供具体操作步骤

---

## 策略选择速查

```
技能状态诊断:
│
├── 行数 >500 且 冗余明显     → 🗑️  A: 精简优先
├── 行数 <200 且 内容单薄     → 📚  B: 丰富优先
├── 格式混乱 / 可读性差       → ✨  C: 美化格式
├── 结构不合理 / 类型不匹配   → 🔧  D: 重构调整
└── 多种问题并存              → 🔀 组合策略
```

---

## 策略 A: 精简优先 (Simplify)

> **适用**: >500 行，冗余内容多，信息密度低
> **目标**: 减少 30-50% 行数，提升信号密度

### 诊断指标

满足以下 **≥3 项** 时启用此策略：

- [ ] 总行数 > 500
- [ ] 有重复内容（相同概念在多处解释）
- [ ] 示例过长（单个示例 >30 行）
- [ ] 段落冗长（单段落 >10 行且非代码）
- [ ] 有"显而易见"的说明（解释 Agent 已知的概念）

### 操作步骤

#### Phase 1: 冗余扫描（5 min）

```markdown
## 冗余检测清单

### 1.1 重复内容检测
- [ ] 同一概念是否在多个章节解释？
- [ ] 示例是否过度相似？
- [ ] 前言区和正文是否有重叠？

### 1.2 低信号内容识别
使用以下标准判断内容是否应该删除：

| 删除理由 | 示例 | 保留条件 |
|---------|------|---------|
| 解释通用概念 | "Git 是一个版本控制工具..." | 技能本身是教 Git 的 |
| 过度详细的背景 | 长篇历史介绍 | 与核心流程直接相关 |
| 冗余示例 | 第 4 个相似的示例 | 展示不同场景/边缘情况 |
| 泛泛提醒 | "注意安全操作" | 具体的 Gotchas |
| 可从上下文推断 | 显而易见的中间步骤 | 有明确验证要求 |

### 1.3 输出：冗余报告
```

#### Phase 2: 精简执行（15 min）

**优先级排序**（按删除收益）：

1. **🗑️ 直接删除**（无价值内容）
   - 重复的说明
   - 通用概念解释
   - 过度背景介绍

2. **📝 压缩改写**（有价值但冗长）
   - 长段落 → 表格/列表
   - 多个相似示例 → 1 个 + 变体说明
   - 冗长步骤 → 合并相关步骤

3. **📦 移至 references/**（有价值但不是核心流程）
   - 详细 API 参考
   - 完整配置选项表
   - 扩展阅读材料

**精简原则**：

```markdown
## 精简操作规范

### 可以删除的
- ❌ 解释 HTTP 是什么（Agent 已知）
- ❌ 第 3 个几乎相同的示例
- ❌ "请确保..." 类泛泛提醒
- ❌ 与其他章节完全重复的内容

### 必须保留的
- ✅ 具体的 Gotchas 坑点
- ✅ 错误处理矩阵
- ✅ 反模式对照表
- ✅ 二进制验证清单
- ✅ 触发明确的引用指针

### 应该移至 references/ 的
- 📦 >100 行的详细参考表格
- 📦 完整 API 文档
- 📦 历史变更记录（超过 3 个版本）
```

#### Phase 3: 验证（5 min）

```markdown
## 精简后验证

### 数量检查
- [ ] 行数减少 ≥20%（目标 30-50%）
- [ ] 所有章节仍存在（除非有意删除）
- [ ] 无死链（移动内容后更新引用）

### 质量检查
- [ ] 核心流程完整性（端到端可执行）
- [ ] 关键信息无丢失
- [ ] 可读性未下降

### 功能检查
- [ ] 重新运行审计脚本，分数不应下降
- [ ] CSO description 未受影响
- [ ] 所有原有功能保留
```

### 精简案例

**Before (冗长)**:

```markdown
## 关于 Git

Git 是一个分布式版本控制系统，由 Linus Torvalds 于 2005 年创建。
它被广泛用于软件开发中，用于跟踪代码变更。Git 具有很多优点，
比如速度快、支持离线工作、分支功能强大等等。

在本技能中，我们将使用 Git 来管理你的提交记录。
你需要先安装 Git...
（后续还有 20 行 Git 入门介绍）
```

**After (精简)**:

```markdown
## 前置依赖

- **Git**: 用于提交操作（[安装指南](https://git-scm.com/downloads)）
- 基础 Git 命令知识（add, commit, push）

> 📖 Git 入门教程不在本技能范围内
```

**效果**: 从 ~25 行 → 4 行，节省 84%

## 策略 B: 丰富优先 (Enrich)

> **适用**: <200 行，内容不足，缺少关键章节
> **目标**: 增加 50-100% 高质量内容

### 诊断指标

满足以下 **≥3 项** 时启用此策略：

- [ ] 总行数 < 200
- [ ] 缺少必备章节（目标/步骤/示例/注意事项任一缺失）
- [ ] 注意事项只有泛泛提醒（无具体 Gotchas）
- [ ] 无错误处理章节
- [ ] 示例过少或过于简单
- [ ] 无验证清单

### 操作步骤

#### Phase 1: 缺口分析（5 min）

```markdown
## 内容缺口扫描

### 必备章节检查
- [ ] 任务目标/定位
- [ ] 操作步骤（清晰可执行）
- [ ] 使用示例（≥1 个端到端）
- [ ] 注意事项/Gotchas（具体而非泛泛）

### 进阶章节检查
- [ ] 错误处理（异常分类+处理方式）
- [ ] 验证清单（二进制通过/不通过）
- [ ] 反模式对照表
- [ ] 边缘情况处理

### 深度检查
- [ ] Happy Path First 排序？
- [ ] CSO description 质量？
- [ ] TDD 验证记录或豁免？

### 输出：缺口优先级列表
```

#### Phase 2: 内容补充（15-20 min）

**按优先级补充**：

**P0 - 必须有**（缺了就不合格）：

```markdown
## 1. 补充任务目标

### 模板
> **定位**: {一句话说清这个技能解决什么问题}
>
> **能做什么**:
> - {功能1}
> - {功能2}
>
> **不能做什么**（明确边界）:
> - {不在范围内}

## 2. 补充操作步骤

### 要求
- 每个步骤可执行、可验证
- Happy Path First（90% 场景在前）
- 包含预期结果

### 模板
1. {具体动作} → {预期结果}
2. {具体动作} → {预期结果}
   - {子步骤或细节}

## 3. 补充示例

### 最小要求
- 至少 1 个端到端示例
- 包含用户输入和期望输出

### 模板
用户: "{真实用户输入}"
你:
1. {响应步骤1}
2. {响应步骤2}
结果: "{最终输出}"
```

**P1 - 应该有**（缺了扣分）：

```markdown
## 4. 升级注意事项为 Gotchas

### 从泛泛到具体

❌ 泛泛版:
> 注意操作安全

✅ 具体版:
> | 坑点 | 后果 | 正确做法 |
> |------|------|---------|
> | `git add .` 会添加所有文件 | 可能提交敏感信息 | 使用 `git add -p` 逐个确认 |

## 5. 补充错误处理

### 模板
| 异常类别 | 触发条件 | 处理方式 | 是否重试 |
|---------|---------|---------|---------|
| 输入错误 | 参数为空/类型不符 | 终止 + 提示正确格式 | 否 |
| 工具调用失败 | CLI exit ≠ 0 | 检查错误输出后重试 | 是(≤3次) |

## 6. 补充验证清单

### 模板
- [ ] {二进制检查项1}: `{如何自动判断}`
- [ ] {二进制检查项2}: `{如何自动判断}`
```

**P2 - 最好有**（锦上添花）：

```markdown
## 7. 补充反模式对照表

## 8. 补充 TDD 记录或豁免说明
```

#### Phase 3: 验证（5 min）

```markdown
## 丰富后验证

### 数量检查
- [ ] 行数增加 50-100%（不过度）
- [ ] 缺失章节已补全

### 质量检查
- [ ] 新增内容符合写作规则（R1-R14）
- [ ] 无填充式废话（每行都有决策价值）
- [ ] 审计分数明显提升

### 平衡检查
- [ ] 不过度丰富（Type 1 不应 >300 行）
- [ ] 如需大量细节，考虑移至 references/
```

## 策略 C: 美化格式 (Beautify)

> **适用**: 格式不统一，可读性差，但内容质量尚可
> **目标**: 统一风格，大幅提升可读性

### 诊断指标

满足以下 **≥3 项** 时启用此策略：

- [ ] 格式风格不统一（混用中英文标点等）
- [ ] 缺少表格化呈现（大段文字应转为表格）
- [ ] 标题层级混乱
- [ ] 代码块/列表使用不规范
- [ ] 空白使用不当（过多或过少）
- [ ] 引用格式不一致

### 操作步骤

#### Phase 1: 格式审计（5 min）

```markdown
## 格式一致性检查清单

### 标题层级
- [ ] 只使用 ## 和 ### （避免 #### 及更深）
- [ ] 同级标题语法一致
- [ ] 标题命名规范（动词优先/名词优先统一）

### 列表与表格
- [ ] 并列项使用表格（>3 项时）
- [ ] 步骤使用有序列表
- [ ] 特征使用无序列表
- [ ] 表格有对齐的列标题

### 代码块
- [ ] 所有代码有语言标识
- [ ] 内联代码使用反引号
- [ ] 命令使用代码块

### 空白与分隔
- [ ] 一级章节间有空行
- [ ] 列表项间无多余空行
- [ ] 重要内容使用分割线

### 特殊格式
- [ ] 重点使用 **粗体**
- [ ] 警告使用 ❌/⚠️/✅ 等 emoji
- [ ] 引用使用 > blockquote
```

#### Phase 2: 格式统一（10-15 min）

**标准化规则**：

```markdown
## 格式规范手册

### 标题规范
- H1 (#): 仅用于文档标题（1个）
- H2 (##): 主要章节（目标/步骤/示例/注意事项）
- H3 (###): 子章节（Phase 1/Phase 2/变体流程）

### 表格使用时机
✅ 使用表格的情况：
- 对比信息（优缺点、前后对比）
- 配置选项（参数/类型/默认值/说明）
- 检查清单（多项是/否判断）
- 状态汇总（维度/权重/得分）

❌ 不使用表格的情况：
- 单纯的步骤列表 → 用有序列表
- 连续的叙述文本 → 保持段落

### 列表使用时机
1. **有序列表 (1. 2. 3.)**: 操作步骤、有顺序的流程
2. **无序列列表 (-)**: 特征列举、并列选项
3. **任务列表 (- [ ])**: 检查清单、待办事项

### Emoji 使用规范
| Emoji | 用途 | 示例 |
|-------|------|------|
| ✅ | 正确/通过/完成 | ✅ 已完成 |
| ❌ | 错误/失败/禁止 | ❌ 不要这样做 |
| ⚠️ | 警告/注意 | ⚠️ 此操作不可逆 |
| 📖 | 参考文档链接 | 📖 详细指南 |
| 💡 | 提示/建议 | 💡 可以考虑... |
| 🚩 | 红旗警告 | 🚩停下来重新开始 |
| 📦 | 文件/目录 | 📦 references/ |
| 🔗 | 链接 | 🔗 相关资源 |

### 代码块规范
\`\`\`{language}
// 代码内容
\`\`\`

常用语言标识:
- bash/shell: 命令行操作
- yaml/yml: Front Matter / 配置
- markdown: Markdown 示例
- python/js/ts: 代码示例
- powershell/ps1: PowerShell 脚本
- text: 无特定语言
```

#### Phase 3: 验证（5 min）

```markdown
## 美化后验证

### 一致性检查
- [ ] 全文格式风格统一
- [ ] 标题层级正确（无跳级）
- [ ] 表格/列表使用恰当

### 可读性检查
- [ ] 快速浏览能抓住重点
- [ ] 关键信息突出显示
- [ ] 无视觉拥挤感

### 工具验证
- [ ] Markdown 渲染正常
- [ ] 代码高亮正确
- [ ] 表格对齐良好
```

## 策略 D: 重构调整 (Restructure)

> **适用**: 结构性问题，类型不匹配，需要重新组织
> **目标**: 使架构更合理，类型匹配

### 诊断指标

满足以下 **≥2 项** 时启用此策略：

- [ ] 当前结构与类型判定不匹配（如 Type 1 却有 skills/）
- [ ] 章节组织不符合逻辑（如注意事项在最前面）
- [ ] 内容分布不合理（核心流程在 references/ 中）
- [ ] 需要升级/降级类型（如 Type 2 → Type 4）
- [ ] 子技能职责划分不清

### 操作步骤

#### Phase 1: 架构评审（10 min）

```markdown
## 当前架构分析

### 类型再判定
当前声明类型: Type __
实际特征:
- 功能数: __ (单一/多模块)
- 主文件行数: __
- 目录结构: {描述}
- references/ 数量: __

再判定结果: Type __
是否匹配? ✅ / ❌

### 章节逻辑审查
当前章节顺序:
1. {章节1}
2. {章节2}
...

建议顺序（Happy Path First）:
1. 目标与范围
2. 核心流程（主路径）
3. 详细步骤
4. 示例
5. 错误处理
6. 验证清单
7. 注意事项/Gotchas
8. 引用索引

### 内容分布审查
SKILL.md 内容:
- 核心流程: {占比}%
- 详细参考: {占比}%
- 示例: {占比}%
- 其他: {占比}%

理想分布（Type N）:
- 核心流程: {理想占比}%
- 详细参考: {应移至 references/}%
- 示例: {理想占比}%
```

#### Phase 2: 重构执行（20-30 min）

**常见重构场景**：

**场景 1: 类型升级**

```
Type 1 → Type 2:
  创建 skills/ 目录
  拆分子技能
  原文件改为协调器

Type 2 → Type 4:
  添加 references/ 目录
  子技能添加自有 references/
  根文件简化为路由器
```

**场景 2: 章节重组**

```markdown
## 章节重组模板

### Step 1: 标记所有章节
当前: ## {章节名} (行 X-Y)

### Step 2: 评估每个章节
| 章节 | 位置合理性 | 内容质量 | 行动 |
|------|-----------|---------|------|
| {章} | ✅合适/❌需移动 | ✅好/❌需改进 | 保留/移动/重写/删除 |

### Step 3: 执行重组
按新顺序重新组织内容
更新内部引用和锚点
```

**场景 3: 内容迁移**

```
SKILL.md → references/:
  - >100 行的详细参考表格
  - 完整 API 文档
  - 历史记录（>3 个版本）
  - 扩展阅读

references/ → SKILL.md:
  - 核心流程步骤（不应藏在 refs 中）
  - 关键决策逻辑
  - 必读的入门指导
```

#### Phase 3: 验证（10 min）

```markdown
## 重构后验证

### 架构验证
- [ ] 新结构符合类型定义
- [ ] 层级深度 ≤ 3
- [ ] 子技能（如有）独立可用

### 内容验证
- [ ] 无内容丢失
- [ ] 所有引用路径有效
- [ ] 核心流程在 SKILL.md 中（不在 refs 中）

### 功能验证
- [ ] 重新运行审计脚本
- [ ] 分数保持或提升
- [ ] 所有原有功能可用
```

## 组合策略

当多种问题并存时，按以下顺序组合执行：

### 推荐组合顺序

```
① D: 重构调整（如有结构性问题）
   ↓ (先解决架构问题)
② A: 精简优先（如内容冗余）
   ↓ (清理后再丰富)
③ B: 丰富优先（如内容不足）
   ↓ (最后统一外观)
④ C: 美化格式（始终最后做）
```

**为什么是这个顺序？**
- 先重构：避免在错误的结构上浪费精力
- 再精简：清理后再知道真正缺什么
- 后丰富：基于清理后的骨架补充
- 最后美化：内容确定后再统一格式

### 组合示例

**案例**: 一个 650 行的 Type 3 技能，格式混乱且有冗余

```
执行计划:
1. D 重构 (20min) → 调整为 Type 4，拆分 references/
2. A 精简 (15min) → 从 650 行减至 400 行
3. B 丰富 (10min) → 补充缺失的错误处理章节
4. C 美化 (10min) → 统一格式风格

总耗时: ~55min
预期效果: 650行 → ~380行, 质量等级 C→A
```
