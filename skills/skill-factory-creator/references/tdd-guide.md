# TDD 驱动技能创建 — 完整指南

> **版本**: v2.1.0

## 核心铁律

```
NO SKILL WITHOUT A FAILING TEST FIRST
```

这条铁律适用于**新技能创建**和**已有技能编辑**。

---

## TDD 三阶段详解

### RED 阶段：观察失败（15-20 min）

#### 目标

在编写任何技能内容之前，先证明"没有这个技能时 Agent 会失败"。

#### 步骤 1：设计压力场景（5 min）

**设计原则**：

| 原则 | 说明 | 示例 |
|------|------|------|
| **真实性** | 模拟真实高压力情况 | "生产环境 API 宕机，每分钟损失 $15,000" |
| **多样性** | 至少 3 种不同压力类型 | 时间紧/权威压/疲劳态/模糊需求 |
| **具体性** | 明确输入和期望输出 | 不能说"处理错误"，要说"遇到 403 时列出权限" |

**压力维度矩阵**：

```
                ┌──────────┬──────────┬──────────┬──────────┐
                │ 时间压力 │ 权威压力 │ 疲劳状态 │ 模糊需求 │
├───────────────┼──────────┼──────────┼──────────┼──────────┤
│ 高            │ "5分钟"  │ "老板说" │ "第10次" │ "大概..."│
│ 中            │ "30分钟" │ "团队决定"│ "第3次"  │ "类似..."│
│ 低            │ 无限制   │ 自己决定 │ 首次     │ 详细需求 │
└───────────────┴──────────┴──────────┴──────────┴──────────┘
```

**推荐场景组合**：

```markdown
### 场景 A: 紧急修复 + 权威压力
Context:
- 生产 API 故障，错误率 100%
- 损失: $15,000/分钟
- Manager: "立即修复！"

Input: 错误日志片段
Expected: 系统性调试流程（非快速 hack）

Pressure factors: [时间紧: 高] [权威: 高] [疲劳: 中]

---

### 场景 B: 模糊需求 + 自主决策
Context:
- 用户说"优化一下这个技能"
- 没有明确说明要优化什么
- 有 3 个可能的优化方向

Input: 一个 500 行的 SKILL.md
Expected: 先澄清需求再动手

Pressure factors: [模糊: 高] [时间: 中] [权威: 低]

---

### 场景 C: 疲劳 + 边缘情况
Context:
- 这是今天第 8 个类似的任务
- 输入数据有异常格式
- 正常流程无法处理

Input: 格式异常的配置文件
Expected: 优雅降级而非崩溃

Pressure factors: [疲劳: 高] [边缘: 高] [时间: 低]
```

#### 步骤 2：运行基线测试（10 min）

**操作方法**：

1. 准备干净的测试环境
2. 使用子代理执行每个压力场景
3. **逐字记录**所有行为

**记录模板**：

```markdown
## 基线测试记录 — 场景 {N}

### 输入
{具体的输入内容}

### 期望行为
{根据技能目标定义的期望}

### 实际行为（逐字记录）
Agent: "{原话输出}"
Action: "{执行的操作}"
Result: "{结果}"

### 违规检测
- [ ] 违反了哪条规则？{规则名}
- [ ] 具体违规行为？{描述}
- [ ] 严重程度？{低/中/高}

### 合理化借口
Agent 说: "{原话借口}"
分析: 这属于哪类合理化策略？
  □ "太简单不需要"
  □ "时间紧跳过"
  □ "我先做再补"
  □ "这是特殊情况"
  □ "精神 vs 字面"
  □ 其他: ___________
```

#### 步骤 3：识别失败模式（5 min）

**汇总分析**：

```markdown
## 失败模式总结

### Top 3 常见违规
1. **{违规类型}**: 出现 {N} 次，占比 {X}%
   - 典型借口: "{最常用语}"
   - 根本原因: {为什么 Agent 会这样}

2. **{违规类型}**: ...

3. **{违规类型}**: ...

### 合理化策略统计
| 策略 | 使用次数 | 成功率 | 应对方案 |
|------|---------|--------|---------|
| "时间紧" | N | X% | {具体禁止规则} |
| "太简单" | N | X% | {具体禁止规则} |
| ... | ... | ... | ... |

### 技能必须解决的核心问题
1. ...
2. ...
3. ...
```

---

### GREEN 阶段：编写最小技能（20-30 min）

#### 目标

只针对 RED 阶段观察到的问题编写技能，不过度设计。

#### 核心原则

| 原则 | 说明 | 反模式 |
|------|------|--------|
| **最小化** | 只解决观察到的具体问题 | 为未观察到的"可能问题"编写规则 |
| **针对性** | 每条规则对应一个记录的违规行为 | 写泛泛的"最佳实践" |
| **不假设** | 不预判 Agent 会怎么犯错 | 写"注意 XXX"而不写具体禁止 |

#### 编写步骤

1. **从失败模式开始**
   - 取 RED 阶段的 Top 3 违规
   - 为每个违规编写明确的禁止规则

2. **添加正面指导**
   - 对每个"不要"，配一个"这样做"
   - 包含具体示例

3. **构建验证清单**
   - 二进制通过/不通过的检查项
   - 覆盖所有关键路径

4. **添加红旗警告**
   - 将常见合理化借口列入 Red Flags 列表

**示例：从失败到规则**

```
RED 阶段发现:
  Agent 在时间压力下会说"太紧急了，先快速修复"
  
GREEN 阶段写入:
  ## 🚩 Red Flags - STOP and Start Over
  
  如果你说以下任何一句话，停下来重新开始：
  - "时间不够做完整调查"
  - "先快速修复，之后再调查"
  - "这是特殊情况"
  
  **所有这些意味着：停下来。遵循完整的调试流程。**
```

---

### REFACTOR 阶段：修补漏洞（15-20 min）

#### 目标

发现 Agent 找到的新漏洞，逐一修补直到稳定。

#### 循环流程

```
┌─────────────────────────────────────────────┐
│           REFACTOR 循环                       │
├─────────────────────────────────────────────┤
│                                             │
│  ① 用当前技能运行压力场景                     │
│       ↓                                     │
│  ② 观察行为，发现新漏洞                      │
│       ↓                                     │
│  ③ 记录漏洞类型和合理化方式                   │
│       ↓                                     │
│  ④ 在 SKILL.md 中添加/修改规则               │
│       ↓                                     │
│  ⑤ 再次测试                                 │
│       ↓                                     │
│  ⑥ 全部通过？→ ✅ 完成                      │
│       ↓ 否                                  │
│  → 回到 ②（最多循环 3 轮）                  │
│                                             │
└─────────────────────────────────────────────┘
```

#### 漏洞分类与应对

| 漏洞类型 | 表现 | 应对策略 |
|---------|------|---------|
| **新合理化方式** | Agent 找到了新的借口 | 添加到红旗警告列表 |
| **规则边界** | Agent 找到了规则的灰色地带 | 收窄规则表述，消除歧义 |
| **规则冲突** | 两条规则互相矛盾 | 明确优先级或合并规则 |
| **遗漏场景** | 新的压力组合暴露问题 | 补充对应的处理规则 |
| **过度遵守** | Agent 过于死板，失去灵活性 | 添加例外条款 |

#### 循环终止条件

满足以下任一条件即可终止 REFACTOR：

- [ ] 连续 2 轮测试无新漏洞
- [ ] 已达 3 轮循环上限（标记为"需持续观察"）
- [ ] 所有核心场景通过，仅边缘情况有微小偏差（可接受）

## 技能类型与测试策略

| 技能类型 | 特征 | 测试方法 | 成功标准 |
|---------|------|---------|---------|
| **规范强制型** | 强制遵守规则 | 学术提问 + 压力场景 + 多重压力组合 | 最大压力下仍遵守规则 |
| **技术方法型** | 教授具体技术 | 应用场景 + 边界情况 + 信息缺失测试 | 能正确应用到新场景 |
| **思维模式型** | 思维模型 | 识别场景 + 反例测试 | 知道何时/如何应用 |
| **参考文档型** | API 文档、命令参考 | 检索场景 + 应用场景 + 空白测试 | 能找到并正确使用信息 |

## TDD 豁免条件

以下情况可以简化或跳过 TDD 流程：

| 豁免条件 | 要求 | 记录方式 |
|---------|------|---------|
| **Type 1 快速路径** | 单功能 + 非关键任务 | 在 Front Matter 添加 `tdd: simplified` |
| **参考文档型技能** | 纯信息检索类 | 在 Front Matter 添加 `tdd: validation-only` |
| **紧急修复** | 生产环境紧急 bug | 事后补测 + 记录豁免原因 |
| **原型/MVP** | 明确标注为实验性 | 在版本号用 `0.x-alpha` + `tdd: deferred` |

**豁免声明模板**：

```yaml
meta:
  tdd: simplified  # Type 1 快速路径，已简化压力测试
  tdd_waiver_reason: "单一功能非关键任务，走快速路径"
  tdd_waiver_date: "2026-05-27"
```

## 常见误区与纠正

| 误区 | 纠正 |
|------|------|
| "TDD 太耗时了" | 不做 TDD = 后期花 10x 时间修复 |
| "我手动测试过了" | 手动 ≠ 自动化，且无法覆盖压力场景 |
| "这个很简单不需要" | 简单代码也会出问题，30 秒就能写完测试 |
| "先写代码再补测试" | 测试后写的 = 不知道该测什么 |
| "TDD 只适用于代码" | TDD 适用于任何流程文档，包括 SKILL.md |

## CSO 触发率评估方法论（Eval Query Design）

> **来源**: 官方 skill-creator (anthropics/skills) / agentskills.io 优化指南
> **目的**: 用结构化的 Eval Query 集合量化评估 description（CSO）的触发准确率
> **与 TDD 的关系**: TDD 验证"技能内容是否正确工作"，CSO Eval 验证"技能是否在正确时机被触发"

### 为什么需要 CSO Eval

TDD 三阶段（RED/GREEN/REFACTOR）解决的是**技能内容质量**问题。但一个再完美的技能，如果 Agent 在用户需要时**不触发它**，就毫无价值。

```
技能价值 = 内容质量(TDD) × 触发准确率(CSO Eval)
         ↑ 已有完整方法论     ↑ 本章节新增
```

### 核心概念

| 概念 | 定义 | 示例 |
|------|------|------|
| **Should-trigger query** | 应该触发该技能的用户输入 | "帮我创建一个新的 AI 技能" → 应触发 skill-factory-creator |
| **Should-not-trigger query** | 不应该触发该技能的用户输入 | "帮我写个 Python 脚本" → 不应触发 skill-factory-creator |
| **Trigger rate** | 触发率 = 触发次数 / 总运行次数 | 0.8 = 80% 概率被触发 |
| **False positive** | 误触发：不该触发却触发了 | 查日志却触发了代码生成技能 |
| **False negative** | 漏触发：该触发却没触发 | 创建技能却没触发 creator |

### Eval Query 设计规范

#### Should-trigger Queries（8-10 个）

从以下 4 个维度设计正例：

| 维度 | 设计要点 | 示例（以 creator 为例） |
|------|---------|----------------------|
| **措辞变化** | 正式/随意/错别字/缩写/中英混杂 | "创建技能" / "建个skill" / "新建skiil" / "make a skill" |
| **明确度变化** | 直接命名 vs 描述需求而不命名 | "用creator做个技能" vs "我需要一个能帮 Claude 学新能力的工具" |
| **细节程度** | 简短 prompt + 重 context prompt | "建个技能" vs (附带详细需求文档) "基于这个需求创建技能" |
| **复杂度变化** | 单步 vs 多步骤嵌入链 | "创建技能" vs "先分析这个工具能不能做成 skill，然后创建它" |

**设计模板**：

```json
{
  "id": "st-01",
  "type": "should-trigger",
  "dimension": "措辞变化",
  "variant": "正式",
  "prompt": "请帮我创建一个新的 AI Agent Skill",
  "expected_trigger": true,
  "notes": "最标准的触发方式，必须通过"
}
```

#### Should-not-trigger Queries（8-10 个）

**最有价值的类型是近误（Near-miss）**——与目标技能共享关键词但实际需要不同技能：

| 近误类型 | 设计方法 | 示例（以 creator 为例） |
|---------|---------|----------------------|
| **同域不同操作** | 共享领域词但动作不同 | "修改这个 SKILL.md" → 应触发 processor 非 creator |
| **同词不同义** | 关键词相同但语义不同 | "写个脚本来自动化" → 不是创建 Skill |
| **泛化请求** | 更宽泛的需求可能匹配多个技能 | "优化我的开发流程" → 不特指创建新技能 |
| **跨技能边界** | 处于两个技能的模糊地带 | "把这个功能封装成可复用的" → 可能是 assembler |

**设计模板**：

```json
{
  "id": "snt-01",
  "type": "should-not-trigger",
  "dimension": "近误: 同域不同操作",
  "variant": "加工请求",
  "prompt": "帮我优化一下现有的 skill-factory-processor",
  "expected_trigger": false,
  "target_skill": "processor",
  "notes": "包含 'skill' 关键词但动作是'优化'非'创建'"
}
```

### Eval Query 套件标准结构

完整的 Eval Query 文件应放在 `evals/evals.json`：

```json
{
  "skill_name": "your-skill-name",
  "description_summary": "一句话描述该技能做什么",
  "created_date": "2026-05-27",
  "queries": {
    "should_trigger": [
      {
        "id": "st-01",
        "prompt": "...",
        "dimension": "措辞变化",
        "variant": "正式"
      },
      {
        "id": "st-02",
        "prompt": "...",
        "dimension": "明确度变化",
        "variant": "间接描述"
      }
    ],
    "should_not_trigger": [
      {
        "id": "snt-01",
        "prompt": "...",
        "dimension": "近误",
        "variant": "同域不同操作",
        "reason": "应触发 {other-skill}"
      }
    ]
  },
  "split": {
    "train_ratio": 0.6,
    "validation_ratio": 0.4
  }
}
```

**数量要求**：

| 类别 | 最少 | 推荐 | 上限 |
|------|------|------|------|
| should-trigger | 8 | 10 | 15 |
| should-not-trigger | 8 | 10 | 15 |
| **总计** | **16** | **20** | **30** |

### 手动评估流程

在没有自动化 eval 引擎的情况下，按以下流程手动执行 CSO 评估：

#### 第一步：准备 Query 套件（15 min）

1. 按 4 维度 × 2 类型设计 20 条 query
2. 标注每条的 dimension 和 variant
3. 按 60%/40% 分为训练集和验证集

#### 第二步：模拟触发判断（20 min）

对每条 query 执行以下思维模拟：

```markdown
## Query 评估记录 — #{id}

**Query**: "{prompt}"
**类型**: should-trigger / should-not-trigger
**维度**: {dimension}

### 模拟过程
1. 我是一个 Agent，看到用户的这句话
2. 我扫描所有已加载技能的 description 字段
3. 我将用户意图与每个 description 匹配...
4. 我的判断: 触发 / 不触发 {skill-name}
5. 判断依据: description 中的关键词 "{...}" 与 用户说的 "{...}" 匹配/不匹配

### 结果
- 预期: {expected}
- 实际判断: {actual}
- 一致性: ✅ 通过 / ❌ 失败
```

#### 第三步：计算指标（5 min）

```
Trigger Rate (正例) = should_trigger 中被触发的数量 / should_trigger 总数
                      目标值: > 0.8 (80%)

False Positive Rate (负例) = should_not_trigger 中被误触发的数量 / should_not_trigger 总数
                            目标值: < 0.2 (20%)

综合得分 = Trigger Rate × 0.6 + (1 - False Positive Rate) × 0.4
         目标值: > 0.85
```

#### 第四步：分析与迭代（可选）

如果综合得分 < 0.85，按以下策略调整 description：

| 问题表现 | 诊断 | 调整方向 |
|---------|------|---------|
| Trigger Rate 低 (< 0.7) | description 太窄或缺少关键触发词 | 补充同义词、增加触发场景列表 |
| False Positive Rate 高 (> 0.3) | description 太宽或触发词太通用 | 收窄范围、增加排除条件、使用更具体的动词组合 |
| 特定维度全失败 | 该维度的表达习惯未覆盖 | 针对该维度补充对应措辞 |
| 训练集通过但验证集失败 | 过拟合到训练 query | 泛化调整，避免写入训练 query 的原文 |

### Description 迭代优化循环

```
┌──────────────────────────────────────────────────┐
│           CSO 迭代优化循环（最多 5 轮）            │
├──────────────────────────────────────────────────┤
│                                                  │
│  ① 编写初始 description（遵循 R9 规则）            │
│       ↓                                          │
│  ② 设计 20 条 Eval Query（10 正 + 10 负）          │
│       ↓                                          │
│  ③ 分割为训练集(60%) + 验证集(40%)                │
│       ↓                                          │
│  ④ 在训练集上手动评估触发率                        │
│       ↓                                          │
│  ⑤ 识别失败案例                                   │
│    - 未触发 → description 太窄                    │
│    - 误触发 → description 太宽                    │
│       ↓                                          │
│  ⑥ 修订 description（泛化而非添加特定关键词）      │
│       ↓                                          │
│  ⑦ 检查 ≤1024 字符限制                            │
│       ↓                                          │
│  ⑧ 在验证集上评估（防过拟合）                       │
│       ↓                                          │
│  ⑨ 记录验证集得分最高的版本                        │
│       ↓                                          │
│  ⑩ 全部通过 或 无显著提升？→ ✅ 完成              │
│       否                                         │
│    → 回到 ④（最多 5 轮）                          │
│                                                  │
└──────────────────────────────────────────────────┘
```

### 快速自检清单

完成 CSO Eval 后，确认以下项目：

- [ ] 正例 ≥ 8 条，覆盖 4 个维度（措辞/明确度/细节/复杂度）
- [ ] 负例 ≥ 8 条，其中近误类 ≥ 4 条
- [ ] 训练集/验证集分割比例约 60/40
- [ ] Trigger Rate > 0.8
- [ ] False Positive Rate < 0.2
- [ ] description 长度仍 ≤ 1024 字符
- [ ] 迭代轮次 ≤ 5 轮
- [ ] 最终版本是验证集最优（非最后版本）

### 与 TDD 的协作关系
