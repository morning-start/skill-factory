# TDD 驱动技能创建 — 完整指南

## 核心铁律

```
NO SKILL WITHOUT A FAILING TEST FIRST
```

这条铁律适用于**新技能创建**和**已有技能编辑**。

---

## TDD 三阶段详解

### RED 阶段：观察失败（15-20 min）

#### 目标

在编写任何技能内容之前，先证明"没有这个技能时 Agent 会失败"。

#### 步骤 1：设计压力场景（5 min）

**设计原则**：

| 原则 | 说明 | 示例 |
|------|------|------|
| **真实性** | 模拟真实高压力情况 | "生产环境 API 宕机，每分钟损失 $15,000" |
| **多样性** | 至少 3 种不同压力类型 | 时间紧/权威压/疲劳态/模糊需求 |
| **具体性** | 明确输入和期望输出 | 不能说"处理错误"，要说"遇到 403 时列出权限" |

**压力维度矩阵**：

```
                ┌──────────┬──────────┬──────────┬──────────┐
                │ 时间压力 │ 权威压力 │ 疲劳状态 │ 模糊需求 │
├───────────────┼──────────┼──────────┼──────────┼──────────┤
│ 高            │ "5分钟"  │ "老板说" │ "第10次" │ "大概..."│
│ 中            │ "30分钟" │ "团队决定"│ "第3次"  │ "类似..."│
│ 低            │ 无限制   │ 自己决定 │ 首次     │ 详细需求 │
└───────────────┴──────────┴──────────┴──────────┴──────────┘
```

**推荐场景组合**：

```markdown
### 场景 A: 紧急修复 + 权威压力
Context:
- 生产 API 故障，错误率 100%
- 损失: $15,000/分钟
- Manager: "立即修复！"

Input: 错误日志片段
Expected: 系统性调试流程（非快速 hack）

Pressure factors: [时间紧: 高] [权威: 高] [疲劳: 中]

---

### 场景 B: 模糊需求 + 自主决策
Context:
- 用户说"优化一下这个技能"
- 没有明确说明要优化什么
- 有 3 个可能的优化方向

Input: 一个 500 行的 SKILL.md
Expected: 先澄清需求再动手

Pressure factors: [模糊: 高] [时间: 中] [权威: 低]

---

### 场景 C: 疲劳 + 边缘情况
Context:
- 这是今天第 8 个类似的任务
- 输入数据有异常格式
- 正常流程无法处理

Input: 格式异常的配置文件
Expected: 优雅降级而非崩溃

Pressure factors: [疲劳: 高] [边缘: 高] [时间: 低]
```

#### 步骤 2：运行基线测试（10 min）

**操作方法**：

1. 准备干净的测试环境
2. 使用子代理执行每个压力场景
3. **逐字记录**所有行为

**记录模板**：

```markdown
## 基线测试记录 — 场景 {N}

### 输入
{具体的输入内容}

### 期望行为
{根据技能目标定义的期望}

### 实际行为（逐字记录）
Agent: "{原话输出}"
Action: "{执行的操作}"
Result: "{结果}"

### 违规检测
- [ ] 违反了哪条规则？{规则名}
- [ ] 具体违规行为？{描述}
- [ ] 严重程度？{低/中/高}

### 合理化借口
Agent 说: "{原话借口}"
分析: 这属于哪类合理化策略？
  □ "太简单不需要"
  □ "时间紧跳过"
  □ "我先做再补"
  □ "这是特殊情况"
  □ "精神 vs 字面"
  □ 其他: ___________
```

#### 步骤 3：识别失败模式（5 min）

**汇总分析**：

```markdown
## 失败模式总结

### Top 3 常见违规
1. **{违规类型}**: 出现 {N} 次，占比 {X}%
   - 典型借口: "{最常用语}"
   - 根本原因: {为什么 Agent 会这样}

2. **{违规类型}**: ...

3. **{违规类型}**: ...

### 合理化策略统计
| 策略 | 使用次数 | 成功率 | 应对方案 |
|------|---------|--------|---------|
| "时间紧" | N | X% | {具体禁止规则} |
| "太简单" | N | X% | {具体禁止规则} |
| ... | ... | ... | ... |

### 技能必须解决的核心问题
1. ...
2. ...
3. ...
```

### GREEN 阶段：编写最小技能（20-30 min）

#### 目标

只针对 RED 阶段观察到的问题编写技能，不过度设计。

#### 核心原则

| 原则 | 说明 | 反模式 |
|------|------|--------|
| **最小化** | 只解决观察到的具体问题 | 为未观察到的"可能问题"编写规则 |
| **针对性** | 每条规则对应一个记录的违规行为 | 写泛泛的"最佳实践" |
| **不假设** | 不预判 Agent 会怎么犯错 | 写"注意 XXX"而不写具体禁止 |

#### 编写步骤

1. **从失败模式开始**
   - 取 RED 阶段的 Top 3 违规
   - 为每个违规编写明确的禁止规则

2. **添加正面指导**
   - 对每个"不要"，配一个"这样做"
   - 包含具体示例

3. **构建验证清单**
   - 二进制通过/不通过的检查项
   - 覆盖所有关键路径

4. **添加红旗警告**
   - 将常见合理化借口列入 Red Flags 列表

**示例：从失败到规则**

```
RED 阶段发现:
  Agent 在时间压力下会说"太紧急了，先快速修复"
  
GREEN 阶段写入:
  ## 🚩 Red Flags - STOP and Start Over
  
  如果你说以下任何一句话，停下来重新开始：
  - "时间不够做完整调查"
  - "先快速修复，之后再调查"
  - "这是特殊情况"
  
  **所有这些意味着：停下来。遵循完整的调试流程。**
```

### REFACTOR 阶段：修补漏洞（15-20 min）

#### 目标

发现 Agent 找到的新漏洞，逐一修补直到稳定。

#### 循环流程

```
┌─────────────────────────────────────────────┐
│           REFACTOR 循环                       │
├─────────────────────────────────────────────┤
│                                             │
│  ① 用当前技能运行压力场景                     │
│       ↓                                     │
│  ② 观察行为，发现新漏洞                      │
│       ↓                                     │
│  ③ 记录漏洞类型和合理化方式                   │
│       ↓                                     │
│  ④ 在 SKILL.md 中添加/修改规则               │
│       ↓                                     │
│  ⑤ 再次测试                                 │
│       ↓                                     │
│  ⑥ 全部通过？→ ✅ 完成                      │
│       ↓ 否                                  │
│  → 回到 ②（最多循环 3 轮）                  │
│                                             │
└─────────────────────────────────────────────┘
```

#### 漏洞分类与应对

| 漏洞类型 | 表现 | 应对策略 |
|---------|------|---------|
| **新合理化方式** | Agent 找到了新的借口 | 添加到红旗警告列表 |
| **规则边界** | Agent 找到了规则的灰色地带 | 收窄规则表述，消除歧义 |
| **规则冲突** | 两条规则互相矛盾 | 明确优先级或合并规则 |
| **遗漏场景** | 新的压力组合暴露问题 | 补充对应的处理规则 |
| **过度遵守** | Agent 过于死板，失去灵活性 | 添加例外条款 |

#### 循环终止条件

满足以下任一条件即可终止 REFACTOR：

- [ ] 连续 2 轮测试无新漏洞
- [ ] 已达 3 轮循环上限（标记为"需持续观察"）
- [ ] 所有核心场景通过，仅边缘情况有微小偏差（可接受）

## 技能类型与测试策略

| 技能类型 | 特征 | 测试方法 | 成功标准 |
|---------|------|---------|---------|
| **规范强制型** | 强制遵守规则 | 学术提问 + 压力场景 + 多重压力组合 | 最大压力下仍遵守规则 |
| **技术方法型** | 教授具体技术 | 应用场景 + 边界情况 + 信息缺失测试 | 能正确应用到新场景 |
| **思维模式型** | 思维模型 | 识别场景 + 反例测试 | 知道何时/如何应用 |
| **参考文档型** | API 文档、命令参考 | 检索场景 + 应用场景 + 空白测试 | 能找到并正确使用信息 |

## TDD 豁免条件

以下情况可以简化或跳过 TDD 流程：

| 豁免条件 | 要求 | 记录方式 |
|---------|------|---------|
| **Type 1 快速路径** | 单功能 + 非关键任务 | 在 Front Matter 添加 `tdd: simplified` |
| **参考文档型技能** | 纯信息检索类 | 在 Front Matter 添加 `tdd: validation-only` |
| **紧急修复** | 生产环境紧急 bug | 事后补测 + 记录豁免原因 |
| **原型/MVP** | 明确标注为实验性 | 在版本号用 `0.x-alpha` + `tdd: deferred` |

**豁免声明模板**：

```yaml
meta:
  tdd: simplified  # Type 1 快速路径，已简化压力测试
  tdd_waiver_reason: "单一功能非关键任务，走快速路径"
  tdd_waiver_date: "2026-05-27"
```

## 常见误区与纠正

| 误区 | 纠正 |
|------|------|
| "TDD 太耗时了" | 不做 TDD = 后期花 10x 时间修复 |
| "我手动测试过了" | 手动 ≠ 自动化，且无法覆盖压力场景 |
| "这个很简单不需要" | 简单代码也会出问题，30 秒就能写完测试 |
| "先写代码再补测试" | 测试后写的 = 不知道该测什么 |
| "TDD 只适用于代码" | TDD 适用于任何流程文档，包括 SKILL.md |
