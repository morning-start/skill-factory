## 1. 项目全景

### 1.1 这是什么？

skill-factory 解决一个很特别的元问题：**如何系统性地创建和管理 AI Agent 技能**。

如果把每个 AI 技能（SKILL.md）看作一个"产品"，那 skill-factory 就是生产这个产品的工厂。它有原料入口（用户需求）、四条生产线（生产/加工/发布/销毁）、质量检验（packager/standardizer），还有一套完整的设计规范（四维分类：轻/重/薄/厚）。

```
用户说"我想做一个技能"
         ↓
    🏭 skill-factory
         ↓
  结构化的 SKILL.md 技能包
```

### 1.2 项目规模

```
总文件: 24 个
总代码: ~6,000 行 Markdown
SKILL.md: 18 个（1根 + 4协调器 + 13执行者）
Reference: 3 个（铁律 / SOP / 架构）
Mermaid: ~107 个图表
```

| 层级 | 数量 | 总行数 | 职责 |
|------|------|--------|------|
| Layer 0（工厂根） | 1 | 225 | 全局入口 |
| Layer 1（协调器） | 4 | 1,017 | 阶段调度 |
| Layer 2（执行者） | 13 | 4,143 | 单一操作 |
| References | 3 | 478 | 详细文档 |

### 1.3 这个项目的独特性

在 AI Agent 生态中，skill-factory 是一种**元技能**（meta-skill）——它不是用来直接回答用户问题的，而是用来**创建其他技能**的。这有点像编译器之于程序：编译器本身也是一个程序，但它的作用是产生其他程序。

这种"自举"属性带来了一个有趣的张力：**skill-factory 自己符不符合它自己定义的标准？** 这个张力贯穿了整个架构分析。

## 2. 三层架构铁律

### 2.1 设计动机

v0.1.0 时代，13 个子技能扁平排列在根目录下，没有任何层次感。哪个技能属于哪个阶段？需要翻文档才能搞清楚。

v0.2.0 引入了三层架构：

```
Layer 0: skill-factory/SKILL.md         ← 1 个（入口）
Layer 1: skills/phase-*/SKILL.md        ← 4 个（协调器）
Layer 2: skills/phase-*/worker/SKILL.md ← 13 个（执行者）
```

**为什么要三层而不是两层？** 两层的问题是：13 个执行者直接挂在根下，像一个 13 个按钮的遥控器——你分不清哪个按钮是"开机"、哪个是"关机"。三层的核心价值是给执行者找到了"归属"：researcher 知道自己是生产流水线的第一站，destroyer 知道自己独自负责销毁阶段。

**为什么不四层？** 这里有个经典的"米勒定律"：人类工作记忆的广度是 7±2。三层的存在让每个文件都能在 3 步之内找到自己在全局中的位置。如果再加一层（比如让 researcher 有自己的子技能），认知成本会指数级上升。

### 2.2 v0.3.0 的"铁律内化"

v0.3.0 做了一件有意思的事：把"最多三层"从一个**设计原则**升级为**可执行的铁律**。具体表现为：

1. **前言区硬编码**：`max_layers_allowed: 3` 字段
2. **独立 SOP 章节**：4 步超三层处理流程（暂停→分析→拆分→确认）
3. **流程嵌入**：planner（层级预检）→ generator（层级合规验证）→ packager（最终层级确认）

这相当于在工厂里装了三道安检。问题是：**这个工厂本身就 6,000 行了，还需要 180 行的 SOP 来约束自己吗？**

### 2.3 现实检验：它自己遵守铁律吗？

```
✅ skill-factory/SKILL.md              = Layer 0
✅ skills/phase-*/SKILL.md             = Layer 1
✅ skills/phase-*/worker/SKILL.md      = Layer 2
✅ references/*.md                     = 不算层级 ✓

结论：完全符合三层铁律
```

但有讽刺的地方：为了描述"如何不超过 3 层"，项目写了 3 个 reference 文件共计 478 行（占项目 8%），其中光 SOP 就 182 行。**维护铁律的成本已经接近铁律本身要保护的价值了**。

## 3. 四阶段流水线

### 3.1 整体设计

```mermaid
flowchart LR
    A[用户需求] --> P1[阶段一: 生产<br/>5 步流水线]
    P1 --> P2[阶段二: 加工<br/>4 种加工器]
    P2 --> P3[阶段三: 发布<br/>3 步顺序]
    P3 --> P4[阶段四: 销毁<br/>按需触发]
    
    style P1 fill:#a5d6a7,stroke:#388e3c,color:#ffffff
    style P2 fill:#ffe0b2,stroke:#fb8c00,color:#e65100
    style P3 fill:#90caf9,stroke:#1976d2,color:#ffffff
    style P4 fill:#ef9a9a,stroke:#c62828,color:#b71c1c
```

四个阶段的工业隐喻映射得很好：

| 阶段 | 工业类比 | 实际含义 |
|------|---------|---------|
| 生产 | 原料→成品 | 从需求生成技能 |
| 加工 | 成品→精加工 | 优化已有技能 |
| 发布 | 质检→出厂 | 版本管理+提交 |
| 销毁 | 退役→回收 | 标记废弃 |

### 3.2 生产阶段：设计最完善的流水线

```mermaid
flowchart LR
    R[researcher] --> A[analyzer] --> PL[planner] --> G[generator] --> K[packager]
```

生产阶段的 5 步流水线是四个阶段中**设计最优秀**的：

- **门禁设计精准**：analyzer 完整度≥80%、packager 评分≥60，两次门禁分别守住"信息是否充足"和"输出是否合格"
- **回调机制有节制**：analyzer 不通过最多回调 researcher 1 次，packager 不通过最多回调 generator 2 轮——"最多"两个字避免了死循环
- **快速路径**：Type 1 技能（轻+薄）跳过加工阶段直接发布，效率提升 85%

### 3.3 加工阶段：策略模式的亮点

加工阶段不采用固定流水线，而是根据技能体量**动态选择策略**：

| 行数 | 策略 | 流程 |
|------|------|------|
| >500 | 精简优先 | simplifier → enricher → standardizer |
| <200 | 丰富优先 | enricher → beautifier → standardizer |
| 200-500 | 均衡 | simplifier → enricher → beautifier → standardizer |

**值得称赞的设计**：循环保护机制。同一技能最多加工 3 轮，行数变化<5%触发警告——这个细节体现了"不要过度优化"的工程哲学。

### 3.4 发布与销毁：功能完整但体量悬殊

发布阶段三步顺序（version→metadata→release）设计干净利落，支持 Type 1 快速模式。销毁阶段逻辑完整，deprecated 模板设计规范。

但这里有一个**失衡**：生产+加工占了项目 80% 的内容和复杂度，发布+销毁占比仅 20%。销毁阶段只有一个执行者（destroyer），但为了符合架构规范，还额外创建了一个协调器文件（phase-destruction/SKILL.md，164 行）。**协调器的存在感几乎为零**——它和 destroyer 的内容大量重复。

## 4. 架构深层问题

### 4.1 问题一：协调器的"价值危机"

四个协调器的设计动机是清晰的：作为阶段入口，承担调度、质量门禁、错误处理。

但实际执行中出现了问题：

| 协调器 | 执行者数 | 协调器行数 | 协调器价值评估 |
|--------|---------|-----------|--------------|
| phase-production | 5 | 262 | ⭐⭐⭐⭐⭐ 调度逻辑复杂，价值高 |
| phase-processing | 4 | 296 | ⭐⭐⭐⭐ 策略模式有独特价值 |
| phase-publishing | 3 | 295 | ⭐⭐⭐ 三步顺序简单，但快速模式有价值 |
| phase-destruction | 1 | 164 | ⭐⭐ **大量内容与 destroyer 重复** |

**根因分析**：当执行者只有 1 个时，协调器本质上是在"转述"，而不是"协调"。这违背了"协调器 = 调度 + 质量门禁"的初衷。生产阶段协调器之所以有价值，是因为 5 个执行者之间有前后依赖和回调机制需要管理；销毁阶段只有一个执行者，协调器只是在前面多写了一层。

### 4.2 问题二：职责边界模糊（packager vs. standardizer）

这是整个项目最隐蔽的设计问题：

| 检查维度 | Standardizer | Packager |
|---------|-------------|----------|
| 前言区完整性 | 检查一（5 字段） | 通用验证（6 字段） |
| 类型专项结构 | 检查五（Type 1-4） | 四种验证模式（Type 1-4） |
| 评分体系 | 4 维度 × 25% | 4 维度 × 25% |
| 评分等级 | 90+/70+/60+/<60 | 90+/70+/60+/<60 |

两套评分维度和等级标准**几乎一模一样**。唯一的区别是：Packager 处于生产阶段末尾（生成后立即验证），Standardizer 处于加工阶段末尾（加工后再次验证）。

**这合理吗？** 表面上合理——就像工厂里有"出厂质检"和"精加工后再检"，但问题是**它们检查的内容完全重叠**。合理的分工应该是：

```
Packager:    结构验证（类型结构、文件完整性、目录层级）——可以自动化
Standardizer: 内容规范（命名、描述长度、链接有效性、格式）——需要人工判断
```

但现在两者都在做"结构+类型+评分"，形成了双重的、几乎相同的工作。

### 4.3 问题三：Enricher 的"职责黑洞"

Enricher 名义上是"内容丰富"，但实际承载了 6 种操作：

1. 示例增强
2. References 拆分（薄→厚）
3. 能力扩展（轻→重）
4. Mermaid 增强
5. **多技能整合（Assembly）** ← 这不应该是单个操作
6. 优化策略诊断

"多技能整合"尤其值得关注。将多个技能合并为一个，需要（1）分析各技能的职责关系，（2）选择整合模式，（3）设计新的协调器，（4）处理依赖关系——这是一个独立于"丰富"的完整能力，不亚于生产阶段的流水线。

如果按照三层架构"单一职责"的原则，Assembly 应该是一个独立的执行者或场景。

### 4.4 问题四：版本号的不一致（已发现的 bug）

| 位置 | 版本号 |
|------|--------|
| 根 SKILL.md | **v0.3.1** |
| 4 个协调器 | v0.2.0 |
| 13 个执行者 | v0.2.0 |
| metadata.json | v0.2.0 |

根文件自己在 v0.3.0→v0.3.1 迭代了（做了 Type 3 拆分），但 17 个子技能和 metadata.json 全部停留在 v0.2.0。这就像一本合集的封面写着"第三版"，翻开里面全是第二版的内容——虽然没有逻辑错误（子技能确实没改），但从项目管理角度是不一致的。

### 4.5 问题五：6,000 行的技能——自己吃得消吗？

```
项目总行数: ~6,000 行 Markdown
Mermaid 图表: ~107 个
SKILL.md 文件: 18 个
平均每个 SKILL.md: ~300 行
```

按照 skill-factory 自己的"四维分类"标准：一个 **Type 4（重+厚）** 的项目，推荐使用"完整路径"（生产→全量加工→发布+监控），预计耗时"5h+"。

但 skill-factory 自己只是一个**文档规范项目**（没有代码逻辑，只有声明和流程描述）。把 6,000 行的规范文档按照"工业工厂"的体量来管理，是否存在**过度工程化（over-engineering）** 的风险？

## 5. 更好的设计

### 5.1 核心问题：工厂隐喻是否太重了？

```
当前设计哲学: 🏭 工业工厂 → 标准化、流水线、质检、SOP
更适合的设计: 🔧 工匠工坊 → 指南、清单、模式、灵活组合
```

skill-factory 的核心价值是什么？**帮助人系统性地创建合格的技能**。这个核心价值不需要 13 个执行者 + 107 个 Mermaid 图表来传达。事实上，最核心的内容可能是：

1. **三层架构的目录约定**（在哪放什么文件）
2. **四维分类的判断方法**（轻/重/薄/厚怎么判定）
3. **SKILL.md 的前言区模板**（标准元数据该怎么写）
4. **几个关键的设计模式**（流水线、策略选择、快速路径）

这些内容在 300-500 行的主文件中完全能讲清楚。当前的 6,000 行是因为把"工厂流水线"的隐喻**当成了真实的系统架构**来建模——每个阶段必须有协调器、每个协调器必须有调度逻辑、每种调度逻辑必须有 Mermaid 图……

### 5.2 改进方案：从 Type 4 到 Type 2（重+薄）

如果让我重新设计 skill-factory：

```
目标定位: 技能创建指南 + 设计模式库（而非"工厂操作系统"）
分类:     重+薄（功能全面，但内容精炼）
规模:     ~1,500 行（精简 75%）
```

**核心精简策略**：

| 当前 | 改进后 | 理由 |
|------|--------|------|
| 13 个独立执行者 SKILL.md | **合并为 4 个阶段指南** | 执行者之间职责重叠、耦合紧密，独立维护成本高 |
| 107 个 Mermaid 图表 | **保留 20 个核心图表** | 大量图表是同质化的流程变体，可压缩 |
| Phase-destruction 协调器 | **合并到 destroyer** | 只有 1 个执行者的协调器没有独立存在价值 |
| Packager 与 Standardizer 双重评分 | **单一评分体系** | 消除重复 |
| Assembly 内嵌于 Enricher | **独立为"整合器"** | Assembly 复杂度已超过单个操作 |

**精简后的结构**：

```
skill-factory/
├── SKILL.md (300行)                     ← 核心指南
├── references/
│   ├── design-patterns.md               ← 流水线/策略/快速路径等模式
│   ├── type-classification.md           ← 四维分类详细
│   └── skill-standards.md               ← 规范检查清单
└── examples/
    ├── type1-simple-skill/SKILL.md      ← Type 1 示例
    └── type4-complex-skill/SKILL.md     ← Type 4 示例
```

### 5.3 保留什么、去掉什么

| 保留 | 原因 |
|------|------|
| ✅ 三层架构铁律 | 核心理念，不可去掉 |
| ✅ 四维分类法 | 实用的判断工具 |
| ✅ 四阶段核心流程 | 操作指南的主线 |
| ✅ 标准前言区模板 | 每天都会用到 |
| ✅ 场景快速路由 | 用户入口 |
| ✅ Type 1 快速路径 | 实用价值高 |

| 精简/合并 | 原因 |
|-----------|------|
| ⚠️ 13 个执行者 → 4 个阶段指南 | 减少 75% 文件数 |
| ⚠️ 4 个协调器 → 合并到阶段指南 | 协调器独立为文件的理由变弱 |
| ⚠️ SOP 180 行 → 20 行摘要 | SOP 使用场景少 |
| ⚠️ Packager 564 行 → 验证清单 | 评分机制过于复杂 |
| ⚠️ Researcher 546 行 → 研究清单 | 13 个 Mermaid 可压缩 |

### 5.4 更有深度的设计：从"工厂"到"模式语言"

一个更有启发性的设计方向：不是让 skill-factory 成为一个需要先学习的"元系统"，而是变成一个**可以随时查阅的模式目录**。

类比——你想写一个 Web 应用，你不是先去读"Web 框架工厂操作手册"，而是：
1. 想快速起项目 → vite 模板
2. 想加路由 → react-router 文档
3. 想加状态 → zustand 文档

同理，用户想创建技能时，不应该先"学习 skill-factory 的四阶段流水线和 13 个执行者职责"，而应该是：
1. 想快速建简单技能 → Type 1 快速模板
2. 想拆复杂技能 → 拆分模式文档
3. 想检查规范 → 规范检查清单

**把"操作系统"变成"速查手册"**，这才是更匹配目标用户（AI Agent）的设计。

## 6. 综合评价

### 6.1 架构健康度

| 维度 | 评分 | 说明 |
|------|------|------|
| **层级规范性** | 9/10 | 三层铁律严格执行，-1 因版本不一致 |
| **设计完整性** | 9/10 | 四阶段覆盖全生命周期，闭环完整 |
| **职责清晰度** | 6/10 | 存在重叠（Standardizer/Packager）和过载（Enricher） |
| **体量适度性** | 5/10 | 6,000 行对规范项目偏重，存在过度工程化迹象 |
| **可维护性** | 7/10 | references/ 拆分合理，但独立文件过多 |
| **文档一致性** | 6/10 | 前言区基本统一，但 dependency 语义不统一，版本不一致 |
| **实用价值** | 8/10 | 核心概念（三层/四维/流水线）有价值，SOP 价值存疑 |

### 6.2 亮点

1. **三层架构铁律** — 不是空泛的"推荐"，而是从元数据到流程都有执行机制。这个"从理念到机制"的设计思路值得学习。
2. **快速路径设计** — Type 1 技能跳过加工阶段的决策，体现了"识别简单场景并简化处理"的务实态度。
3. **循环保护机制** — 加工阶段的行数变化<5%警告 + 3 轮上限，避免了"无限优化"的反模式。
4. **四维分类法（轻/重/薄/厚）** — 把模糊的"技能复杂程度"变成了可量化、可讨论的二维坐标系。

### 6.3 核心建议（优先级排序）

| 优先级 | 建议 | 工作量 | 收益 |
|--------|------|--------|------|
| 🔴 P0 | 统一版本号至 v0.3.1 | 5 分钟 | 消除显著不一致 |
| 🔴 P0 | 分离 Standardizer 与 Packager 职责 | 30 分钟 | 消除核心设计重叠 |
| 🟡 P1 | 合并 Phase-destruction 协调器至 destroyer | 20 分钟 | 减少冗余 |
| 🟡 P1 | 将 Assembly 从 Enricher 独立为 assembler | 1 小时 | 单一职责 |
